How to Run

cd ~/Desktop/Artificial\ Intelligence/emf-parent && claude --agent feature-pipeline

Claude Code reads .claude/agents/feature-pipeline.md automatically. No setup script needed. The pipeline initializes jdtls (LSP) for code traversal in Phase 0.

Mode 1 (Full Pipeline) now includes an optional multi-epic coordination step. Answer "yes" at input #8 to enable the coordinator, which auto-checks the shared registry at 6 checkpoints.

Mode 2 (Resume) validates all paths from pipeline-state.json against the current machine. If a teammate cloned the repo on a different machine, the pipeline asks for updated paths before continuing.

Mode 3 (Jump) detects which phases already have artifacts and starts from the first missing one. It auto-generates session-memory.md from provided artifacts if not supplied.

Mode 5 (Decompose) is for architects. It reuses existing pipeline skills — /ba for BRD analysis, /cross-repo-tracer for codebase scanning, /architect + /designer for interface design, and /coordinate for registry publishing. It runs these skills across all epics at once to identify shared modules, design contracts, assign ownership, and publish to the registry. Run once before developers start.

For New Teammates

  1. Clone the repo — the .claude/ directory comes with it (skills, agents, CLAUDE.md, principles.md)
  2. Install Superpowers: claude install-plugin superpowers
  3. (Optional) Install jdtls: brew install jdtls — enables LSP-based code traversal instead of grep fallback
  4. Run: claude --agent feature-pipeline
  5. (Multi-epic) If working on a BRD with other developers:
    • Architect runs Mode [5] first — identifies shared modules, publishes to registry
    • Each developer runs Mode [1] and selects "yes" for multi-epic coordination
    • The pipeline auto-coordinates via the shared-modules-registry

That is it. No environment variables, no config files, no Docker. The pipeline reads everything from .claude/.

Phase Start Announcement

Every phase begins with a mandatory announcement showing which skill(s) will run, the model used, and a one-line description of what the phase does. This makes the pipeline transparent.

Pause Prompt

After every phase, the pipeline shows: artifacts produced, session memory updates, git snapshot tag, any blockers, and available commands (continue, skip, revert N, status, resolve, exit).

No File Copying

The pipeline reads BRD, code repos, and UI screenshots from their original locations. It never copies external repos into emf-parent. Only binary formats (PDF/DOCX) get text extracted to brd-raw.md. Paths are stored in pipeline-state.json.

Live HTML Dashboard

live-dashboard.html is created at Phase 0 (if the user opts in) and updated after every phase by the orchestrator. Open in browser at any time to see current pipeline state.

Each update includes: progress bar, phase summary with key numbers, Mermaid diagrams, data tables (Q&A, API contracts, findings), sidebar quick links, and stats bar. At Phase 12, the dashboard is finalized as the feature blueprint HTML.

The orchestrator updates the dashboard — subagents do not. This is non-negotiable in the execution order.

Multi-Epic Coordination

When a BRD has multiple epics assigned to different developers, the pipeline coordinates shared modules to prevent duplicate implementations. The core philosophy: minimize dependencies while tasking. AI agents make sequencing practical because the wait is hours, not weeks.

Module Status Lifecycle

stateDiagram-v2 [*] --> designed : Decomposer creates designed --> claimed : Developer claims claimed --> in_progress : First commit in_progress --> ready_for_review : PR opened, tests pass ready_for_review --> merged : PR merged to main in_progress --> reverted : PR reverted reverted --> in_progress : Re-merge in_progress --> designed : Abandoned (30d auto-release) note right of designed : Interface defined, no owner note right of merged : Consumers use real code note right of reverted : Consumers fall back to mocks

Staleness Lifecycle

Active (commits within 7 days) → Stale (7-14 days) → WARN owner → Abandoned (14-30 days) → GitHub issue to owner → Auto-released (30+ days) → status reverts to "designed", anyone can re-claim.

Branching Strategy

Single shared branch per ticket. All developers commit to raidlc/<ticket>. No v1/v2/v3 proliferation. Always git pull --rebase before push.

Developer Identification Flow

  1. Developer runs Mode [1] with multi-epic enabled
  2. Pipeline finds raidlc/*/epic-division branch in kalpavriksha
  3. Reads epic-assignment.json — shows team list
  4. "Who are you?" — developer selects from list (verified against git config)
  5. Loads assignment: epics, shared modules to build/consume, layer
  6. Checks out raidlc/<ticket> in code repos (creates if not exists, pulls if exists)
  7. Injects epic package constraints into session memory
  8. Pipeline starts with full context

Push Protocol

Before every push at any phase:

git fetch origin raidlc/<ticket>
git rebase origin/raidlc/<ticket>
# If conflict → interactive resolution
# If clean → push

Phase tags are per-epic: raidlc/<ticket>/{epic}/phase-01 so developers' tags don't overwrite each other.

Mechanism 3: Confidence-Based Escalation

Any claim below C4 in any phase output is surfaced to the user with the evidence gap and a request to verify or accept risk.

Calibrated Confidence Scale (C1-C7) — Cross-Cutting

principles.md is a cross-cutting concern enforced in every phase, every skill, every subagent — not just the Critic. It is applied at 3 levels:

Level 1: CLAUDE.md Rule 6 (Global)

Tells Claude about the C1-C7 framework, reversibility matrix, and anti-patterns. Applied to ALL conversations.

Level 2: Every SKILL.md (16 skills)

Each skill has a "Reasoning Principles" section right after frontmatter: "Read .claude/principles.md at phase start."

Level 3: Every Subagent Prompt (Pipeline orchestrator)

A compact Principles Injection Block is included in every Agent call — C1-C7 scale, doubt resolver, anti-patterns. Subagents don't need to read the full file.

1. Every claim carries C1-C7 confidence level

6 Safety Rules

  1. Always confirm before executing — never auto-revert
  2. Always show file-level impact — user sees exactly what changes
  3. Git tags are force-updated — re-running a phase updates its tag
  4. Session memory cleanup is surgical — only entries from reverted phases removed (by _(Phase)_ suffix)
  5. Rework log is preserved — reverts logged in process-log.md
  6. Never revert past branch creation — the raidlc branch start is the hard floor

Circuit Breaker

If the same phase pair has cycled 3 times, the pipeline stops and escalates:

CIRCUIT BREAKER — Phase 11 (Reviewer) has routed back to Phase 9 (Developer) 3 times. This suggests a systemic issue, not a simple fix.

Options: [A] Try one more cycle. [B] Accept current state with known issues. [C] Revert to an earlier phase.

Why This Approach

MECE skills — no overlap, no gaps. 16 active skills are mutually exclusive and collectively exhaustive. Each skill owns exactly one responsibility. No two skills produce the same output. Deprecated skills (/prd-generator, /gap-analyser) are merged, not duplicated.
Incremental session memory survives context compaction. Session memory is updated after every decision, not batched at phase end. If Claude's context compacts mid-phase, the last decision is already persisted. Each entry is tagged with its source phase for surgical cleanup during revert.
Cross-repo tracer prevents blind spots. Phase 5 maps write/read paths across all 5 repos before any design begins. Any claim of "0 modifications needed" requires C6+ evidence from reading actual code. No more discovering missing Thrift changes in code review.
Backend readiness gate catches production issues before review. Phase 9b checks N+1 queries, Thrift backward compatibility, cache invalidation, resource management, error handling, and migration safety. Issues found here are cheaper to fix than in production.
Model optimization — opus for thinking, sonnet for coding, haiku for mechanical. Opus handles irreversible judgment calls (BA, Architect, Critic). Sonnet handles coding tasks where it benchmarks highest (Developer, Designer). Haiku handles template output (Blueprint, Build Verify). This optimizes cost, speed, and quality.
Auto-updating live dashboard. The live-dashboard.html is updated by the orchestrator after every phase — not by subagents. Open in browser anytime to see current state with progress bars, Mermaid diagrams, Q&A history, and API contracts.
3-option revert (Full / Artifacts / Re-run). Full revert restores codebase to exact state after target phase. Artifacts-only keeps code but re-runs analysis. Re-run overwrites from target phase onward. All with surgical session memory cleanup.
Circuit breaker prevents infinite rework loops. If the same phase pair cycles 3 times, the pipeline stops and escalates. This prevents Developer-Reviewer ping-pong and forces the team to address systemic issues.
Path validation lets any teammate resume. Mode 2 (Resume) validates all paths from pipeline-state.json against the current machine. If a teammate cloned the repo on a different machine, the pipeline asks for updated paths. No manual JSON editing needed.
3 question mechanisms ensure nothing is silently assumed. Subagent question collection surfaces below-C5 uncertainties. 6 mandatory checkpoints pause the pipeline for user review. C4 escalation catches low-confidence claims in any phase output.
Every claim is evidence-backed (C1-C7). The confidence scale forces agents to cite code, not assume. A claim without evidence is C1 (speculative). Reversibility determines action threshold.
Critic + Compliance catch what authors miss. Phase 2 runs two parallel agents — Critic challenges logic using adversarial questioning, Analyst (compliance mode) verifies claims against actual code.
Migrator plans AND validates. Phase 6b plans migrations with expand-then-contract enforcement. Phase 9d validates that the Developer followed the plan. Both check GUARDRAILS G-05.4.
20+ documents generated automatically. BA, PRD, contradictions, gap analysis, UI requirements, grooming questions, blocker decisions, code analysis per repo, cross-repo trace, architect, analyst, migrator, designer, QA, developer, backend readiness, compliance, migration validation, SDET, reviewer, process log, approach log, session memory, blueprint HTML, live dashboard.
Resumable from any phase. Pipeline state saved to JSON + git tags. Smart restart warns if codebase changed. Path validation handles different machines. Mode 3 (Jump) auto-detects which phases need running.
Reviewer gap routing — re-run, manual, or accept-risk. Each finding from the Reviewer gets classified with three options: route back to Developer, fix manually outside the pipeline, or accept the risk (logged in approach-log). No binary pass/fail.
