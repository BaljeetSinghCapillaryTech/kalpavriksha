## 1. Current Behaviour

### 1.1 Tier & Program Platform (docs.capillarytech.com)

- **Tiers (Slabs)**: Loyalty program levels with eligibility criteria, upgrade/downgrade logic, renewal conditions. One tier strategy per program. KPI types: Current Points, Lifetime Points, Lifetime Purchases, Tracker Value.
- **Tier Renewal**: Conditions based on purchase threshold, visits, points earned, or trackers. Evaluation logic: Any(N-1), All, or Custom expression.
- **Tier Downgrade**: Options include one-tier-below, to-appropriate-tier, or to-lowest-tier.
- **Subscription/Partner Programs**: Exist today as "supplementary partner programs" -- membership programs linked to a loyalty program. Currently configured through a fragmented, low-fidelity interface.

### 1.2 Existing Database Schema (MySQL -- warehouse)

| Table | Purpose | Key Fields |
|-------|---------|-----------|
| `partner_programs` | Core subscription program definition | id, org_id, loyalty_program_id, type (EXTERNAL/SUPPLEMENTARY), name, description, is_active, is_tier_based, points_exchange_ratio, expiry_date, backup_partner_program_id |
| `partner_program_enrollment` | Member enrollment (external partner programs) | customer_id, partner_program_id, is_linked, current_slab_id, current_slab_expiry_date, enrolled_on |
| `supplementary_partner_program_enrollment` | Member enrollment (supplementary programs) | customer_id, partner_program_id, is_linked, is_active, membership_start_date, membership_end_date |
| `partner_program_slabs` | Tier hierarchy within partner programs | partner_program_id, serial_number, name |
| `partner_program_tier_sync_configuration` | Maps partner program slabs to loyalty program slabs | partner_program_slab_id, loyalty_program_slab_id |
| `supplementary_membership_history` | Lifecycle event log | source (LINKING/DELINKING/UPDATE/...), action (STARTED/RENEWED/EXPIRED/...), additional_action (SLAB_UPGRADE/RENEWAL/NO_ACTION) |
| `supplementary_membership_cycle_details` | Subscription duration config | cycle_type (DAYS/MONTHS), cycle_value |
| `supplementary_partner_program_expiry_reminder` | Expiry reminders | days_in_expiry_reminder, communication_property_values |

### 1.3 Existing Codebase

- **emf-parent**: `PointsEngineRuleConfigThriftImpl.createOrUpdatePartnerProgram(PartnerProgramInfo, programId, orgId)` -- existing Thrift endpoint for partner program CRUD.
- **intouch-api-v3**: `UnifiedPromotion` MongoDB document model with maker-checker (parentId/version pattern). Used as reference pattern for subscription program implementation.
- **api/prototype**: Extended Fields model with `EntityType` enum (CUSTOMER, TRANSACTION, LINEITEM, LEAD, COMPANY, CARD, USERGROUP2). Price will be handled as an Extended Field, not a first-class subscription field.

## 2. Problem Statement

Subscription programs exist in Capillary today but are configured through a fragmented, low-fidelity interface. There is:
- No unified listing view with status, pricing, subscriber counts, and benefits at a glance
- No comparison across subscription plans
- No meaningful benefit-to-subscription linkage visible to program managers
- No AI assistance for configuration
- No maker-checker approval workflow native to the platform
- Benefits attached to subscriptions are invisible at setup time -- marketers must navigate a separate promotions module

With paid subscriptions now live and multi-tier subscription logic emerging, the configuration surface must be rebuilt to match the quality standard of the Tiers module. This epic delivers a first-class Subscriptions module inside the Garuda platform.

---

## 3. Goals

1. **Unified Listing**: Program managers can view all subscription programs from a single listing page with status, subscriber counts, pricing, and benefits visible at a glance.
2. **Guided Configuration**: Create and edit subscription programs through a structured multi-step form with end-to-end validation.
3. **Generic Maker-Checker**: Every config change creates a pending record for approval before going live. The maker-checker is reusable for future entities (Tiers, Benefits).
4. **Lifecycle Management**: Full state machine (Draft, Pending Approval, Scheduled, Active, Paused, Expired, Archived) with future-dated enrollment support.
5. **API-First**: Self-reliant APIs with validation logic in the API layer. Both UI and future aiRa integration call the same endpoints.

## 5. User Stories

### US-01: Subscription Listing View (E3-US1)

**As a** program manager,
**I want to** see all subscription programs for my loyalty program from a single listing view,
**so that I can** assess their status, subscriber counts, pricing, and benefits at a glance.

**Acceptance Criteria:**

- [ ] AC-01: GET /subscriptions returns paginated list of subscriptions for a program with: name, description, status, benefits count, subscriber count, group tag, last modified date/by
- [ ] AC-02: Header stats show: Total Subscriptions, Active count, Scheduled count, Total Subscribers (aggregate across all active subscriptions)
- [ ] AC-03: Status filter supports multi-select: Active, Draft, Scheduled, Paused, Expired
- [ ] AC-04: Search supports case-insensitive free-text across subscription name and description
- [ ] AC-05: Sorting supported on: Subscribers (desc default), Last Modified, Name
- [ ] AC-06: Grouped view organizes subscriptions by group_tag
- [ ] AC-07: Benefits modal returns all benefits linked to a subscription, with tier indicator for tier-based subscriptions
- [ ] AC-08: Row actions available: Edit, Duplicate, Deactivate/Archive, View Change Log (change log deferred -- future)

### US-02: Subscription Create & Edit (E3-US2)

**As a** program manager,
**I want to** create or edit a subscription program through a guided form,
**so that I can** configure the program efficiently with full validation.

**Acceptance Criteria:**

**Step 1 -- Program Basics:**
- [ ] AC-09: Name (required), Description (optional), Duration (required: numeric + unit DAYS/MONTHS/YEARS)
- [ ] AC-10: Subscription Type toggle: Tier-Based / Non-Tier (default: Non-Tier)
- [ ] AC-11: Validation: Name and Duration required before advancing
- [ ] AC-12: Duplicate action pre-populates form with cloned data, name appended with "(Copy)", saved as Draft

**Step 2 -- Expiry & Additional Information:**
- [ ] AC-13: Program-level expiry date (optional): overrides individual enrollment duration when set
- [ ] AC-14: Restrict to one active program per member (toggle): powered by EMF ENABLE_PARTNER_PROGRAM_LINKING setting
- [ ] AC-15: Migrate on expiry: dropdown -- "No migration" / "Migrate to program [X]" (select target program)
- [ ] AC-16: If Tier-Based: Linked Tier selector (required, dropdown of program's active tiers)
- [ ] AC-17: If Tier-Based: Tier Downgrade on Exit toggle -- when ON, requires downgrade target tier selection

**Step 3 -- Benefits:**
- [ ] AC-18: Non-Tier subscription: flat list of available benefit templates. Program manager selects which to include.
- [ ] AC-19: Tier-Based subscription: benefits pre-filtered by linked tier's configured benefit categories
- [ ] AC-20: Each benefit card shows: name, category type, configured value, active state
- [ ] AC-21: Add/remove benefit actions available

**Step 4 -- Reminders:**
- [ ] AC-22: Up to 5 reminders with: days before expiry (numeric) + channel (SMS/Email/Push)
- [ ] AC-23: Timeline visualization showing when reminders fire relative to expiry date

**Step 5 -- Custom Fields:**
- [ ] AC-24: 3 levels of custom fields: META (program-level), LINK (enrollment), DELINK (unenrollment)
- [ ] AC-25: Custom field picker: multi-select from org-level defined extended fields, filtered by field type compatibility
- [ ] AC-26: PAUSE and RESUME level fields supported for programs with pause-enabled

**Form Actions:**
- [ ] AC-27: Save as Draft: saves to MongoDB with DRAFT status, no maker-checker
- [ ] AC-28: Submit for Approval: changes status to PENDING_APPROVAL, routes to approval queue
- [ ] AC-29: Cancel: discards unsaved changes with confirmation dialog
- [ ] AC-30: Edit of ACTIVE subscription: if maker-checker enabled, creates a pending version (new MongoDB doc with parentId pointing to ACTIVE). Live version remains active until approved.

### US-03: Generic Maker-Checker (Cross-Cutting)

**As a** platform,
**I want** a generic maker-checker flow that any entity can plug into,
**so that** configuration changes require approval before going live, and the same logic works for subscriptions now and tiers/benefits later.

**Acceptance Criteria:**

- [ ] AC-31: State machine: DRAFT -> PENDING_APPROVAL -> ACTIVE (approve) or DRAFT (reject)
- [ ] AC-32: Edit-of-ACTIVE: creates new DRAFT doc with parentId + version increment. ACTIVE doc stays live.
- [ ] AC-33: On APPROVE of versioned edit: ACTIVE doc becomes SNAPSHOT, DRAFT doc becomes ACTIVE.
- [ ] AC-34: On REJECT: DRAFT status reverts to DRAFT (not deleted). Comments preserved.
- [ ] AC-35: Review endpoint: POST /subscriptions/approvals with approvalStatus (APPROVE/REJECT) + comment
- [ ] AC-36: List pending approvals: GET /subscriptions/approvals returns all PENDING_APPROVAL subscriptions
- [ ] AC-37: Generic implementation: maker-checker logic lives in a shared package, not hardcoded to subscriptions. Subscriptions are the first consumer. Entity-specific hooks (pre-approve, post-approve) are pluggable.
- [ ] AC-38: Publish-on-approve: On APPROVAL, full subscription state is written from MongoDB to MySQL (partner_programs, cycle_details, tier_sync_configuration, expiry_reminders). No MySQL writes during DRAFT/PENDING_APPROVAL lifecycle.

### US-04: Lifecycle Management (E3-US4)

**As a** program admin,
**I want to** manage the lifecycle states of subscription programs,
**so that I can** pause, reactivate, and schedule programs from the configuration UI.

**Acceptance Criteria:**

**Lifecycle States:**

| State | Description | Transitions To | Visible To |
|-------|-------------|---------------|-----------|
| Draft | Saved but not published | Active, Archived | Admin only |
| Pending Approval | Submitted for maker-checker review | Active (on approval), Draft (on rejection) | Admin + Approver |
| Scheduled | Approved but start date is in the future | Active (on start date) | All admins |
| Active | Live. Members can enroll, renew, redeem benefits. | Paused, Expired, Archived | All admins |
| Paused | Temporarily suspended. Existing enrollments maintain benefits, new enrollments blocked. | Active | All admins |
| Expired | Past program-level expiry date. New enrollments blocked. Existing enrollments complete their cycle. | Archived | All admins |
| Archived | Permanently deactivated. Read-only history preserved. | None | Admin (read-only) |

- [ ] AC-39: Pause action: changes Active to Paused. New enrollments blocked, existing retain benefits.
- [ ] AC-40: Resume action: changes Paused back to Active.
- [ ] AC-41: Archive action: permanently deactivates. Read-only.
- [ ] AC-42: Scheduled state: subscription with future start date. Activation job runs at midnight to transition to Active.

**Future-Dated Enrollment:**
- [ ] AC-43: Link API supports membershipStartDate in the future. Creates enrollment in PENDING state.
- [ ] AC-44: PENDING enrollment does not activate benefits until membershipStartDate is reached.
- [ ] AC-45: Activation job runs nightly: transitions PENDING enrollments with membershipStartDate <= today to ACTIVE.
- [ ] AC-46: Reschedule: calling Link API again with new future date cancels old PENDING, creates new PENDING atomically.
- [ ] AC-47: Past date rejection: Link API returns 400 ERR_INVALID_DATE for past membershipStartDate.

**Tier Downgrade on Exit:**
- [ ] AC-48: For tier-based subscriptions with Tier Downgrade on Exit enabled: when subscription expires or member is unenrolled, member's tier changes to the configured downgrade target. TIER_DOWNGRADE event fired.

### US-05: API Contract (E3-US5)

**Subscription Program Configuration APIs (intouch-api-v3):**

| Endpoint | Method | Purpose |
|----------|--------|---------|
| /subscriptions | GET | List all subscription programs for a program with stats, filtering, sorting |
| /subscriptions | POST | Create a new subscription program. Returns validation diff before confirming. |
| /subscriptions/{id} | GET | Get a single subscription program with full configuration |
| /subscriptions/{id} | PUT/PATCH | Edit subscription config. If maker-checker on and ACTIVE, creates pending version. |
| /subscriptions/{id}/status | PUT | Change lifecycle status: pause, resume, archive |
| /subscriptions/{id}/benefits | GET/POST | List or link benefits to a subscription program |
| /subscriptions/approvals | GET/POST | Maker-checker: list pending approval items; approve or reject a pending change |

**Enrollment APIs (existing in api/prototype, extended):**

| Endpoint | Method | Purpose |
|----------|--------|---------|
| POST v2/partnerProgram/linkCustomer | POST | Enroll member. Supports future membershipStartDate for PENDING state. |
| POST v2/partnerProgram/deLinkCustomer | POST | Unenroll member. Can target PENDING or ACTIVE enrollment via updateType. |
| POST v2/partnerProgram/customerPartnerProgramUpdate | POST | Update tier for tier-based subscription. Pause/resume actions. |
| GET v2/partnerProgram/customerActivityHistories | GET | Member's subscription history, benefit usage, lifecycle events. |

## 6. Data Architecture

### 6.1 Dual Storage Pattern (Publish-on-Approve)

```
                     DRAFT / PENDING_APPROVAL phase
                     ===============================
                     MongoDB (subscription_programs collection)
                       - Full subscription config
                       - Reminders, custom fields, group_tag
                       - Benefits references (benefit_ids)
                       - parentId / version (maker-checker)
                       - Status: DRAFT | PENDING_APPROVAL

                                    |
                              ON APPROVE
                                    |
                                    v

                     ACTIVE (published) phase
                     ========================
                     MySQL (warehouse tables):
                       - partner_programs (core fields)
                       - supplementary_membership_cycle_details (duration)
                       - partner_program_tier_sync_configuration (tier linkage)
                       - supplementary_partner_program_expiry_reminder (reminders)
                     
                     MongoDB (subscription_programs collection):
                       - Status updated to ACTIVE
                       - Remains source of truth for metadata fields
                         not in MySQL (group_tag, custom_fields, etc.)
```

### 6.3 MySQL Tables (Existing -- Reused on Publish)

- `partner_programs`: core fields (name, description, is_active, is_tier_based, expiry_date, type=SUPPLEMENTARY)
- `supplementary_membership_cycle_details`: cycle_type, cycle_value
- `partner_program_tier_sync_configuration`: partner_program_slab_id <-> loyalty_program_slab_id
- `partner_program_slabs`: slab hierarchy (used for tier-based subscriptions)
- `supplementary_partner_program_expiry_reminder`: days_in_expiry_reminder, communication_property_values

### 6.4 Extended Fields (Price)

Price is NOT a first-class field on the subscription entity. It is an Extended Field created per brand/org using the existing Extended Fields infrastructure in `api/prototype`. The EntityType enum may need a new value (e.g., `PARTNER_PROGRAM` or `SUBSCRIPTION`) to support extended fields on subscription programs.
