## 2. Test Scenarios

### Group A — Subscription CRUD and Listing

**TS-01 — Listing happy path** `[AC-01]`
- **Setup**: 3 subscriptions for org A (status: ACTIVE, DRAFT, PAUSED). Each has at least 1 benefit linked.
- **Action**: GET /v3/subscriptions?page=0&size=10
- **Expected**: Page of 3 subscriptions. Each row includes: `name`, `description`, `status`, `benefitsCount` (computed from benefitIds array length), `subscriberCount` (from Thrift), `groupTag`, `lastModifiedDate`, `lastModifiedBy`.
- **Verify**: Response is paginated (`totalElements`, `page`, `size`). Subscriber count is non-null (may be 0).
- **Risk note**: benefitsCount computed client-side from array length — no N+1 query needed. [C6]

**TS-02 — Header stats aggregation** `[AC-02, KD-44]`
- **Setup**: Org with 5 subscriptions: 2 ACTIVE, 1 DRAFT, 1 PAUSED, 1 ACTIVE with future startDate ("Scheduled").
- **Action**: GET /v3/subscriptions (includes header stats block).
- **Expected**: `headerStats.total=5`, `headerStats.active=2`, `headerStats.scheduled=1` (ACTIVE with future startDate), `headerStats.totalSubscribers` = sum of per-program counts from Thrift bulk call.
- **Verify**: Exactly 1 Thrift call `getSupplementaryEnrollmentCountsByProgramIds` with all program IDs. Not N per row. [G-04.1]

**TS-03 — Status multi-select filter** `[AC-03]`
- **Action**: GET /v3/subscriptions?status=ACTIVE,DRAFT
- **Expected**: Only ACTIVE and DRAFT subscriptions returned. PAUSED/ARCHIVED excluded.
- **Verify**: MongoDB query `{ orgId: X, status: { $in: ["ACTIVE", "DRAFT"] } }`.

**TS-04 — Case-insensitive name/description search** `[AC-04]`
- **Setup**: Subscription named "Gold Membership Program".
- **Actions**: Search with "gold", "GOLD", "membership", "prog".
- **Expected**: All 4 searches return the subscription.
- **Verify**: MongoDB uses `$regex` with `$options: 'i'` or text index.

**TS-05 — Sort ordering** `[AC-05]`
- **Setup**: 3 subscriptions with different subscriber counts, modification dates, names.
- **Actions**: Default sort (subscribers desc), then `sort=lastModified`, then `sort=name`.
- **Expected**: Each sort order returns subscriptions in correct sequence.
- **Edge**: Subscriptions with equal subscriber count — secondary sort key must be deterministic (e.g., by objectId).

**TS-06 — Grouped view by group_tag** `[AC-06]`
- **Setup**: Subscriptions A/B with `groupTag="VIP"`, subscription C with `groupTag=null`.
- **Action**: GET /v3/subscriptions?grouped=true
- **Expected**: Response groups A and B together under "VIP". C appears ungrouped or in a default group.
- **Verify**: No NPE when groupTag is null. [G-02.1]

**TS-07 — Benefits modal** `[AC-07, AC-20, ADR-04]`
- **Setup**: Subscription with `benefitIds=[101, 102, 103]`. Subscription is Tier-Based with linkedTierId set.
- **Action**: GET /v3/subscriptions/{id}/benefits
- **Expected**: 3 benefit objects returned with `name`, `categoryType`, `configuredValue`, `activeState`. Tier indicator present.
- **Edge**: If one benefit (102) has been deleted from benefits service → response should gracefully skip or return stub (no 500). [TS-80 covers dangling reference]

**TS-08 — Row actions availability by state** `[AC-08]`
- **Action**: Check row actions for DRAFT, ACTIVE, PAUSED, ARCHIVED subscriptions.
- **Expected**: DRAFT → Edit, Duplicate. ACTIVE → Edit (creates versioned draft), Duplicate, Deactivate. PAUSED → Resume, Archive, View. ARCHIVED → View only (no Edit, no Deactivate).
- **Verify**: ARCHIVED subscription: PUT returns 422 `InvalidSubscriptionStateException`.

### Group D — SAGA and Failure Scenarios

**TS-41 — SAGA: Thrift failure → status stays PENDING_APPROVAL** `[ADR-03]`
- **Setup**: Mock Thrift to throw `PointsEngineRuleServiceException`.
- **Action**: POST approval.
- **Expected**: MongoDB status remains `PENDING_APPROVAL`. `mysqlPartnerProgramId=null`. No MySQL row created. Exception caught and mapped to 500 (or a retriable error response).
- **Verify**: Subscription can be re-submitted for approval after transient failure.

**TS-42 — SAGA: Thrift success + MongoDB update failure → retry resolves** `[ADR-03, RF-6]`
- **Setup**: Thrift succeeds → returns `partnerProgramId=1001`. Simulate MongoDB update failure (e.g., replica lag / timeout).
- **Step A**: `mysqlPartnerProgramId=1001` persisted in MongoDB before full SAGA commit attempt (idempotency anchor).
- **Step B**: Retry: `mysqlPartnerProgramId` is non-null → skip Thrift call → proceed only with MongoDB `status=ACTIVE` update.
- **Expected**: Final state → ACTIVE, no duplicate MySQL row. [C5 — retry logic verified from ADR-03 design, not yet from code]

**TS-43 — SAGA idempotency: double approval** `[ADR-03, G-06.1]`
- **Action**: POST approval twice (e.g., accidental double-click or retry).
- **Expected**: First call → ACTIVE, `mysqlPartnerProgramId=1001`. Second call detects `mysqlPartnerProgramId` already set → returns 200/idempotent (or 409 if already ACTIVE). NO second Thrift call. NO second `partner_programs` row.

**TS-44 — No MySQL write during DRAFT lifecycle** `[ADR-01]`
- **Actions sequence**: Create subscription → Save as Draft → Edit draft → Submit for approval → (stay in PENDING).
- **Verify at each step**: MySQL `partner_programs` table has NO row for this subscription's `partner_program_identifier`. `mysqlPartnerProgramId` in MongoDB = null.
- **Confidence**: [C7] ADR-01. Thrift call only happens inside `SubscriptionApprovalHandler.onApprove()`.

**TS-45 — MySQL written exactly once on APPROVE** `[ADR-01]`
- **Action**: Approve subscription.
- **Verify**: Exactly 1 row in `partner_programs` for this subscription. Exactly 1 row in `supplementary_membership_cycle_details`. `supplementary_partner_program_expiry_reminder` has 0 rows (ADR-06).
