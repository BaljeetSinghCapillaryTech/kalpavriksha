## Domain Terminology
| Term | Definition | Source |
|------|-----------|--------|
| Tier | A level in a loyalty program hierarchy (e.g., Silver, Gold, Platinum). Members move between tiers based on KPI thresholds. | BRD S1 |
| Slab | Internal name for a tier in the emf-parent codebase (ProgramSlab entity). | Codebase |
| Benefit | A reward or entitlement attached to a tier -- e.g., points multiplier, free shipping, birthday bonus. BRD treats these as first-class objects, not promotions. | BRD S7.2 |
| Benefit Category | A metadata grouping layer for benefit instances. Categories define type (Welcome Gift, Earn Points, etc.), trigger event, and tier applicability. | BRD Epic 4 |
| Benefit Instance | A configured benefit linked to a specific category and tier, carrying the actual reward value. | BRD Epic 4 |
| KPI | Key Performance Indicator used to evaluate tier eligibility -- e.g., Lifetime Points, Lifetime Spend, Transaction Count. | BRD S7.1 |
| Maker-Checker | Approval workflow where config changes create a pending record that must be approved before going live. | BRD S7.1-US4 |
| aiRa | Capillary's conversational AI assistant, embedded in the configuration UI as a side panel. Handles complex structural changes. | BRD S8 |
| Simulation Mode | Feature allowing users to preview the impact of a config change on current member distribution before publishing. | BRD S7.1-US6 |
| Program | A loyalty program belonging to an org. Contains tiers, benefits, subscriptions. | BRD S1 |
| Subscription | A membership program (tier-based or non-tier) with duration, pricing, benefits, and lifecycle states. | BRD Epic 3 |
| Context Layer | Structured JSON representation of program state that aiRa needs to be intelligent about configuration. | BRD S8.1 |

## Codebase Behaviour
| Behaviour | Evidence | Confidence |
|-----------|----------|------------|
| Extended Fields are entity-type-scoped via ExtendedField.EntityType enum in api/prototype. Current types: CUSTOMER, *_TRANSACTION, *_LINEITEM, LEAD, COMPANY, CARD, USERGROUP2. Each type maps to a MongoDB collection (e.g., customer_extended_fields). No PARTNER_PROGRAM type exists yet. | ExtendedField.java:70-110 in api/prototype | C7 |
| Extended field values stored in MongoDB (ExtendedFieldValidationRuleDaoMongoImpl), configs managed via ExtendedFieldFacade -> ExtendedFieldConfigService. REST API at /v2/entity/extendedFieldConfig (GET/POST). | ExtendedFieldResource.java, ExtendedFieldFacade.java in api/prototype | C7 |
| SubscriptionV2Facade in api/prototype handles COMMUNICATION CHANNEL subscriptions (email/SMS opt-in), NOT partner program subscriptions. These are completely different domains despite sharing the "subscription" name. | SubscriptionV2Facade.java in api/prototype | C7 |
| Partner programs have NO REST APIs in intouch-api-v3. All operations go through Thrift: partnerProgramLinkingEvent, partnerProgramDeLinkingEvent, partnerProgramUpdateEvent in emf.thrift. | emf.thrift, grep of intouch-api-v3 controllers | C6 |
| Maker-checker infrastructure exists in intouch-api-v3 via RequestManagementController (/v3/requests/{entityType}/{entityId}/status) and RequestManagementFacade. EntityType enum has: PROMOTION, TARGET_GROUP, STREAK, LIMIT, LIABILITY_OWNER_SPLIT, WORKFLOW, JOURNEY, BROADCAST_PROMOTION. Currently only PROMOTION is routed. Pattern: add new EntityType value + new domain facade + wire in RequestManagementFacade. | EntityType.java, RequestManagementFacade.java, RequestManagementController.java in intouch-api-v3 | C7 |
| Partner programs only have is_active boolean (no lifecycle state machine). No PartnerProgramStatus enum exists in emf-parent. BRD requires 7 states: Draft, Pending Approval, Scheduled, Active, Paused, Expired, Archived. Resolved by KD-07: lifecycle managed in MongoDB, not MySQL. | PartnerProgram.java in emf-parent, partner_programs.sql in cc-stack-crm | C7 |
| UnifiedPromotion pattern in intouch-api-v3: @Document(collection="unified_promotions") with objectId + unifiedPromotionId (UUID). Metadata nested object holds status (PromotionStatus enum: DRAFT, ACTIVE, PAUSED, PENDING_APPROVAL, STOPPED, SNAPSHOT, LIVE, UPCOMING, COMPLETED, PUBLISH_FAILED), orgId, dates, createdBy. Repository extends MongoRepository with @Query methods for org-scoped + status-scoped lookups. | UnifiedPromotion.java, UnifiedPromotionRepository.java, Metadata.java in intouch-api-v3 | C7 |
| UnifiedPromotion maker-checker: edit ACTIVE creates new DRAFT document (version N+1) with parentId pointing to ACTIVE. Submit for approval -> PENDING_APPROVAL. Approve -> old ACTIVE becomes SNAPSHOT, new doc becomes ACTIVE + publishes to EMF via EntityOrchestrator/PromotionTransformerImpl. Reject -> back to DRAFT. StatusTransitionValidator uses EnumMap<CurrentStatus, Set<PromotionAction>> for valid transitions. | UnifiedPromotionFacade.java:createVersionedPromotion(), changePromotionStatus(), StatusTransitionValidator.java | C7 |
| Multi-tenant MongoDB: EmfMongoConfig configures separate emfMongoTemplate via EmfMongoTenantResolver (per-org database). @EnableMongoRepositories routes UnifiedPromotionRepository to emfMongoTemplate. | EmfMongoConfig.java in intouch-api-v3 | C7 |
| PromotionAction enum: SUBMIT_FOR_APPROVAL, APPROVE, REJECT, PAUSE, STOP, RESUME, REVOKE. Each has getNormalizedAction() for validation consistency. RequestManagementController at /v3/requests/{entityType}/{entityId}/status accepts EntityType path var + PromotionStatus query param + StatusChangeRequest body. | PromotionAction.java, RequestManagementController.java in intouch-api-v3 | C7 |

## Key Decisions
| # | Decision | Rationale | Phase | Reversible? |
|---|----------|-----------|-------|-------------|
| KD-01 | cc-stack-crm included as a code repo for DB schema reference | Contains DB schema, tables, and indexes needed by all code agents | Phase 0 | Yes |
| KD-02 | **SUPERSEDED** -- Original scope (E1+E4+E2 parts) replaced by KD-03 | Scope correction by user | Phase 0 | N/A |
| KD-03 | Scope: E3 (Subscription Programs) is PRIMARY. Includes: Subscriptions CRUD, maker-checker for subscriptions, subscription-benefits linking with dummy benefit objects. OUT OF SCOPE: E1 (Tier Intelligence), E4 (Benefit Categories -- real implementation deferred, dummy objects used). | User corrected scope -- subscriptions are the focus for this pipeline run | Phase 1 | No -- scope is locked for this run |
| KD-04 | User stories in scope: E3-US2 (Create/Edit CRUD), E3-US4 (Lifecycle Management), E3-US5 (API Contract) + maker-checker. OUT: E3-US1 (Listing UI), E3-US3 (aiRa). Full REST API layer. Benefit linking uses dummy benefit objects. | Backend-focused run; no UI rendering, no AI assistant integration | Phase 1 (BA Q1) | No |
| KD-05 | Tier-based subscriptions use existing codebase model: own slab hierarchy via partner_program_slabs, synced to loyalty tiers via PartnerProgramTierSyncConfiguration. Do NOT implement BRD's direct tier-linking model. | Keep backward compatibility with existing partner program infra; avoid breaking enrolled members | Phase 1 (BA Q2) | No -- structural decision |
| KD-06 | Subscription pricing uses existing Extended Fields mechanism (api/prototype repo), NOT new DB columns. A 'price' named extended field will be created for SUPPLEMENTARY partner programs. Extended fields are stored in MongoDB, defined per entity type via ExtendedField.EntityType enum. Current entity types: CUSTOMER, TRANSACTION, LINEITEM, LEAD, COMPANY, CARD, USERGROUP2. A new PARTNER_PROGRAM entity type will need to be added to support pricing fields on subscriptions. Extended field configs are org-scoped with per-field settings (mandatory, updatable, PII, etc.). | Reuse existing infrastructure; avoid schema changes to partner_programs table; pricing is metadata not core entity data; extended fields already have validation, audit trail, and API support | Phase 1 (BA Q3) | Yes -- could add DB columns later if extended fields prove insufficient |
| KD-07 | **[ADR] MongoDB-first architecture for subscriptions.** All subscription CRUD, lifecycle states (Draft, Pending Approval, Scheduled, Active, Paused, Expired, Archived), maker-checker workflow, and benefit linking stored in MongoDB (intouch-api-v3), following the UnifiedPromotion pattern. MySQL (emf-parent/partner_programs) ONLY updated when subscription transitions to Active, via EMF Thrift APIs. Implementation: (1) New UnifiedSubscription @Document in intouch-api-v3, (2) New SubscriptionRepository extends MongoRepository, (3) Add SUBSCRIPTION to EntityType enum, (4) Extend RequestManagementFacade routing, (5) New SubscriptionFacade, (6) SubscriptionStatusTransitionValidator for subscription-specific states (SCHEDULED, EXPIRED, ARCHIVED not in promotions), (7) On APPROVE -> call EMF Thrift to create/update MySQL partner_programs record, (8) NO schema changes to partner_programs table. Key pattern from UnifiedPromotion: objectId (Mongo auto) + unifiedPromotionId (immutable business UUID); Metadata nested object with status, orgId, dates; edit ACTIVE creates new DRAFT doc (version N+1) with parentId; StatusTransitionValidator uses EnumMap; EmfMongoConfig with EmfMongoTenantResolver for multi-tenant per-org DB routing. | User decision: proven pattern, no MySQL migration risk, flexible MongoDB doc for nested config, maker-checker versioning already works | Phase 1 (BA Q4) | No -- fundamental architecture |
| KD-08 | **[ADR] Benefits are a first-class entity; subscriptions store only foreign key references.** The subscription MongoDB document stores `benefitIds: ["id1", "id2"]` -- a plain array of benefit IDs. No embedded benefit snapshots, no benefit metadata, no separate subscription_benefits collection. Benefits will be built as their own entity in a future pipeline run (E2/E4) with their own CRUD, maker-checker, and audit trail. They will be mappable to Tiers, Subscriptions, and other entities. At read time, benefit details can be resolved by calling the benefits service with the stored IDs. For this run, dummy benefit IDs are used as placeholders -- the API accepts and stores them without validation against a real benefits service. | User decision: clean separation of concerns; benefits are not a subscription sub-entity but a cross-cutting first-class entity; avoids coupling subscription schema to benefit schema; future-proof for E2/E4 pipeline run | Phase 1 (BA Q5) | Yes -- could embed benefit snapshots later if read performance demands it |
| KD-09 | Reminder config and custom field config both stored in the MongoDB subscription document. Reminders: embedded array `reminders: [{daysBefore: N, channel: "SMS/EMAIL/PUSH"}, ...]` (max 5). Custom fields: embedded config `customFields: {meta: [fieldId, ...], link: [fieldId, ...], delink: [fieldId, ...], pause: [fieldId, ...], resume: [fieldId, ...]}` referencing extended field IDs. This run stores CONFIGURATION ONLY -- no reminder triggering logic, no custom field enforcement at enrollment time. Triggering and enforcement are future enhancements. | Config storage is trivial document fields; keeps API contract complete per BRD E3-US5; avoids external system integration (comm-server) in this run; custom field values at enrollment time handled by existing extended fields API | Phase 1 (BA Q6) | Yes -- could remove fields from doc if not needed |
| KD-10 | **Enrollment API boundary**: (1) api/prototype v2 linking APIs (`v2/partnerProgram/linkCustomer`, `deLinkCustomer`, `customerPartnerProgramUpdate`, `customerActivityHistories`) remain UNCHANGED and untouched. (2) intouch-api-v3 gets NEW v3 subscription CRUD + enrollment APIs that call EMF Thrift DIRECTLY (same pattern as UnifiedPromotion calling PointsEngineRulesThriftService). NOT proxying through api/prototype. (3) MongoDB stores subscription config, lifecycle, maker-checker, benefit IDs. (4) MySQL (via EMF Thrift) stores enrollment records, activation writes. The v3 controller is the unified surface for subscription management -- config AND enrollment -- while api/prototype v2 continues to serve existing callers. | User decision: follows UnifiedPromotion pattern where v3 controller calls Thrift directly; avoids proxying through api/prototype; clean separation: v3 is the new surface, v2 is legacy-compatible; enrollment records stay in MySQL because EMF owns that data | Phase 1 (BA Q7) | No -- defines the service boundary |
