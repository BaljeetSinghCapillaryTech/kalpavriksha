## 1. Executive Summary

This analysis covers **Epic 3 -- Subscription Programs** from the Tiers & Benefits PRD v3. The scope is limited to backend API implementation: subscription CRUD, lifecycle management, maker-checker workflow, benefit ID linking, and the REST API contract. No UI, no aiRa integration.

The fundamental architecture decision is **MongoDB-first** (KD-07): all subscription configuration, lifecycle state machine, maker-checker versioning, and benefit linking live in MongoDB (intouch-api-v3), following the proven UnifiedPromotion pattern. MySQL (emf-parent/partner_programs) is only updated when a subscription transitions to Active state, via EMF Thrift APIs.

## 4. Architecture Decisions

### ADR-1: MongoDB-First Architecture (KD-07)

**Decision**: All subscription config, lifecycle, maker-checker stored in MongoDB (intouch-api-v3). MySQL only updated on Active transition via EMF Thrift.

**Context**: The BRD requires 7 lifecycle states (Draft, Pending Approval, Scheduled, Active, Paused, Expired, Archived). Current MySQL `partner_programs.is_active` is a boolean with no state machine. Adding a status column requires schema migration across all environments.

**Alternatives considered**:
- (a) Add `status` VARCHAR column to MySQL alongside `is_active` -- expand-then-contract migration
- (b) Replace `is_active` with `status` -- breaking change to all callers
- (c) Separate `partner_program_lifecycle` MySQL table -- query complexity

**Chosen**: MongoDB-first, following the UnifiedPromotion pattern that already exists in intouch-api-v3 with proven maker-checker, versioning, multi-tenant routing.

**Consequences**:
- No MySQL schema changes needed
- Subscription config is a rich MongoDB document (flexible, nested)
- Maker-checker versioning (edit ACTIVE creates new DRAFT doc with parentId) comes for free
- EMF Thrift is called only on activation -- enrollment stays in MySQL
- Two sources of truth: MongoDB for config/lifecycle, MySQL for enrollment data

### ADR-2: Benefits as Foreign Key References Only (KD-08)

**Decision**: Subscription document stores `benefitIds: ["id1", "id2"]`. No embedded benefit metadata.

**Context**: Benefits will be a first-class entity in a future pipeline run (E2/E4) with own CRUD, maker-checker, audit trail.

**Consequences**:
- Clean separation of concerns
- Benefit details resolved at read time by calling benefits service
- For this run, dummy benefit IDs accepted without validation

### ADR-3: Enrollment via v3 APIs Calling EMF Thrift Directly (KD-10)

**Decision**: New v3 subscription APIs in intouch-api-v3 call EMF Thrift directly. Existing api/prototype v2 linking APIs remain untouched.

**Context**: The BRD lists enrollment endpoints. The existing v2 APIs work through api/prototype -> EMF Thrift. The new v3 controller follows the UnifiedPromotion pattern of calling Thrift directly from intouch-api-v3.

**Consequences**:
- Unified v3 API surface for subscription management
- v2 APIs continue serving existing callers
- No changes to api/prototype

### ADR-4: Pricing via Extended Fields (KD-06)

**Decision**: Subscription pricing uses the existing Extended Fields mechanism (api/prototype), not new DB columns.

**Context**: Extended fields are stored in MongoDB, defined per entity type via `ExtendedField.EntityType` enum. A new `PARTNER_PROGRAM` entity type will be added.

**Consequences**:
- No schema changes to `partner_programs` table
- Pricing is stored as extended field values (amount, currency)
- Requires adding `PARTNER_PROGRAM` to `ExtendedField.EntityType` enum in api/prototype

## 6. Subscription MongoDB Document Structure

Based on the BRD's 5-step create/edit flow and all key decisions:

```
UnifiedSubscription {
  // MongoDB auto-generated
  objectId: String (@Id)
  
  // Immutable business identifier (UUID, generated on create)
  unifiedSubscriptionId: String
  
  // Core metadata (following Metadata pattern from UnifiedPromotion)
  metadata: {
    name: String (required, unique per org)
    description: String (optional)
    orgId: Long (set from auth context)
    programId: String (loyalty program ID)
    status: SubscriptionStatus (DRAFT, PENDING_APPROVAL, SCHEDULED, ACTIVE, PAUSED, EXPIRED, ARCHIVED)
    createdOn: Date
    lastModifiedOn: Date
    createdBy: Long
    lastModifiedBy: Long
  }
  
  // Step 1: Program Basics
  subscriptionType: String (TIER_BASED | NON_TIER)
  duration: {
    value: Integer (required)
    unit: String (DAYS | MONTHS | YEARS)
  }
  price: {
    amount: BigDecimal (optional -- null means free)
    currency: String (ISO 4217, e.g., "MYR")
  }
  
  // Step 2: Expiry & Additional Info
  expiryDate: Date (optional, program-level expiry)
  restrictToOneActivePerMember: Boolean (default: false)
  migrateOnExpiry: {
    enabled: Boolean
    targetSubscriptionId: String (null = "No migration")
  }
  linkedTierId: String (required if TIER_BASED, null if NON_TIER)
  tierDowngradeOnExit: {
    enabled: Boolean (only for TIER_BASED)
    downgradeTargetTierId: String
  }
  
  // Step 3: Benefits
  benefitIds: List<String> (plain array of benefit ID references)
  
  // Step 4: Reminders (config only, no triggering)
  reminders: [
    { daysBefore: Integer, channel: String (SMS | EMAIL | PUSH) }
  ] (max 5)
  
  // Step 5: Custom Fields (config only, no enforcement)
  customFields: {
    meta: List<Long> (extended field IDs)
    link: List<Long> (extended field IDs)
    delink: List<Long> (extended field IDs)
    pause: List<Long> (extended field IDs)
    resume: List<Long> (extended field IDs)
  }
  
  // Maker-checker versioning
  parentId: String (points to ACTIVE doc when this is a DRAFT created from edit)
  version: Integer (starts at 1, increments on edit-of-active)
  comments: String (review comment during approval/rejection)
  
  // Group tag (BRD mentions group filtering)
  groupTag: String (optional, admin-defined)
}
```

## 7. Lifecycle State Machine

### 7.1 States

| State | Description | Enrollments Allowed? | Benefits Active? |
|-------|-------------|---------------------|-----------------|
| DRAFT | Saved but not published | No | No |
| PENDING_APPROVAL | Submitted for maker-checker review | No | No |
| SCHEDULED | Approved but start date is in the future | No | No |
| ACTIVE | Live program | Yes | Yes |
| PAUSED | Temporarily suspended | No (new blocked) | Yes (existing retain) |
| EXPIRED | Past program-level expiry date | No | Yes (existing complete cycle) |
| ARCHIVED | Permanently deactivated | No | No (read-only history) |

### 7.2 Valid Transitions

| From | Action | To |
|------|--------|-----|
| DRAFT | SUBMIT_FOR_APPROVAL | PENDING_APPROVAL |
| DRAFT | ARCHIVE | ARCHIVED |
| PENDING_APPROVAL | APPROVE | ACTIVE (or SCHEDULED if future start date) |
| PENDING_APPROVAL | REJECT | DRAFT |
| SCHEDULED | (system: start date reached) | ACTIVE |
| ACTIVE | PAUSE | PAUSED |
| ACTIVE | ARCHIVE | ARCHIVED |
| ACTIVE | (system: expiry date reached) | EXPIRED |
| PAUSED | RESUME | ACTIVE |
| EXPIRED | ARCHIVE | ARCHIVED |

### 7.3 Maker-Checker Flow

1. Admin creates subscription -> saved as DRAFT in MongoDB
2. Admin clicks "Submit for Approval" -> status becomes PENDING_APPROVAL
3. Approver reviews -> APPROVE: status becomes ACTIVE (+ Thrift call to create MySQL record) or REJECT: status reverts to DRAFT
4. Admin edits ACTIVE subscription -> new DRAFT document created (version N+1) with parentId pointing to ACTIVE
5. Admin submits edit for approval -> PENDING_APPROVAL
6. Approver approves -> old ACTIVE becomes snapshot/archived, new doc becomes ACTIVE (+ Thrift call to update MySQL record)

## 11. Risk Register

| # | Risk | Severity | Mitigation |
|---|------|----------|-----------|
| R-01 | Two sources of truth (MongoDB for config, MySQL for enrollment) could diverge if Thrift call fails during activation | HIGH | Implement retry + reconciliation; log activation failures; provide admin API to re-trigger activation |
| R-02 | Extended field PARTNER_PROGRAM entity type addition in api/prototype could affect other entity types if not scoped correctly | MEDIUM | Add entity type as enum value only; no changes to existing field configs |
| R-03 | StatusTransitionValidator for subscriptions has more states than promotions (SCHEDULED, EXPIRED, ARCHIVED vs STOPPED) -- reuse vs clone decision | LOW | Clone with subscription-specific states rather than making the promotion validator generic |
| R-04 | Subscription document in MongoDB could grow large if many custom fields, reminders, benefits are configured | LOW | Document size will be well under MongoDB's 16MB limit; typical subscription has <20 benefit IDs, 5 reminders, <50 custom field IDs |
