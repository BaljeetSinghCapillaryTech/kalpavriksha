## Current Behaviour

### Tier (Slab) Configuration
- Tiers are internally called **"Slabs"** throughout the codebase (`ProgramSlab`, `SlabUpgradeService`, `SlabDowngradeService`, `SlabChangeSource`).
- `ProgramSlab` is a JPA entity in the `program_slabs` MySQL table with fields: id, programId, serialNumber, name, description, createdOn, metadata.
- Richer tier config (eligibility thresholds, renewal conditions, downgrade logic) lives in MySQL **strategy tables** (`strategies` table with `strategy_type` classification).
- Tier evaluation runs through the EMF engine via Thrift (`EMFThriftServiceImpl`). Instructions: `SLAB_UPGRADE`, `SLAB_DOWNGRADE`, `SLAB_RENEWAL`.
- Tier upgrade modes: `EAGER` (upgrade first, then points), `DYNAMIC` (proportional), `LAZY` (points first, then upgrade).
- Downgrade logic uses strategy pattern: `FixedTierDowngradeDateCalculator`, `CyclicTierDowngradeDateCalculator`, `FixedCustomerRegistrationTierDowngradeDateCalculator`, `SlabUpgradeBasedTierDowngradeDateCalculator`.
- Renewal conditions map to retention metrics in code: `retentionAmount`, `retentionPoints`, `retentionTracker`, `retentionVisits` in `SlabDowngradeProfileImpl`.
- Tier events are Kafka-published: `TierUpgradeHelper` → `TierUpgraded`, `TierRenewedHelper` → `TierRenewed`.
- **No dedicated REST API** exists for tier CRUD. Configuration is managed through backend services directly.
- **No maker-checker** exists for tier configuration changes. Changes go directly to strategy/config tables.

### Benefits
- Benefits are currently delivered through the **V3 Promotions engine** (`PromotionConfig`, `UnifiedPromotion`).
- `BenefitsAwardedStats` tracks benefit awards per customer (an awards ledger, not a catalog).
- `BenefitType` enum: REWARDS, COUPONS, BADGES, TIER_UPGRADE, TIER_DOWNGRADE, TIER_RENEWAL, ENROL, OPTIN, PARTNER_PROGRAM, etc.
- There is **no standalone benefit entity** — promotions that award benefits serve as the benefit catalog.
- Benefit types in the BRD (Points Multiplier, Flat Points Award, Free Shipping, etc.) are delivery mechanics within REWARDS promotions, not named entity types.

### Maker-Checker (Reference Pattern)
- `UnifiedPromotion` in intouch-api-v3 implements maker-checker for promotions via:
  - Status lifecycle: DRAFT → PENDING_APPROVAL → ACTIVE (approve) / DRAFT (reject)
  - Versioning: `parentId` (links draft to active parent), `version` (incremented on edit), `comments` (reviewer notes)
  - On approval of an edit: active parent → SNAPSHOT, draft → ACTIVE
  - Concurrency: `@Lockable` with distributed lock (TTL 300s)
  - Always on — no toggle to disable
  - No notifications (email/Kafka) on approval/rejection
  - Status transitions enforced by `StatusTransitionValidator`

### Multi-Tenancy
- All core entities use `orgId` as part of composite PKs. Tenant isolation is enforced at the data model level.

## Problem Statement

Program managers (Maya persona) spend ~45 minutes creating a single tier because configuration is fragmented across 6+ screens with no unified API surface. There is no way to create, edit, or manage tiers and benefits through a clean API contract. There is no approval workflow for tier configuration changes — changes go live immediately without review, creating compliance and operational risk. Benefits are invisible as standalone entities, buried within the V3 Promotions system.

This iteration builds the **backend API foundation**: Tier CRUD, Benefit CRUD, and Maker-Checker workflow — all as REST APIs in emf-parent using MongoDB, following the UnifiedPromotion maker-checker pattern from intouch-api-v3.

---

## Goals

1. **Tier CRUD APIs** — Provide REST endpoints to list, create, and update tier configurations with full validation, enabling both direct UI and future aiRa integration to consume the same API surface.
2. **Benefit CRUD APIs** — Establish benefits as first-class entities with their own lifecycle, independent of the V3 Promotions engine, with standalone endpoints for create, list, and update.
3. **Tier-Benefit Linkage** — Allow benefits to be linked to tiers with per-tier parameter values, stored as part of the tier configuration.
4. **Maker-Checker Workflow** — Implement a DRAFT → PENDING_APPROVAL → ACTIVE approval lifecycle for both tier and benefit configuration changes, following the UnifiedPromotion pattern.
5. **API-First Contract** — All validation lives in the API, not in frontend form logic. Both direct UI and aiRa call the same endpoints and receive the same structured validation feedback.

## Scope

### In Scope
- **Tier CRUD**: GET /tiers (list), POST /tiers (create), PUT/PATCH /tiers/{tierId} (update)
- **Benefit CRUD**: GET /benefits (list), POST /benefits (create), PUT/PATCH /benefits/{benefitId} (update)
- **Tier-Benefit Linkage**: Tiers contain/reference linked benefit data
- **Maker-Checker**: Submit for approval, approve, reject — for both tiers and benefits
- **Maker-checker config flag**: Defaults to `true` (always on), hardcoded for now, DB-configurable later
- **All tier config fields**: Name, description, color, eligibility KPI type, eligibility threshold, validity period, renewal condition, renewal schedule, downgrade target, downgrade schedule, grace period, upgrade schedule (SlabUpgradeMode), upgrade bonus, nudge/communication config, validate-downgrade-on-return-transaction toggle
- **All benefit fields**: Name, type (Points Multiplier, Flat Points Award, Coupon Issuance, Badge Award, Free Shipping, Custom), category, trigger event, description, parameters, tier linkage with per-tier values
- **MongoDB storage** for both tier and benefit configuration documents
- **Structured validation errors** — field-level, human-readable error responses (not HTTP 500)
- **Multi-tenancy** — orgId scoping on all endpoints, following existing platform patterns
- **Backend only** — emf-parent (kalpavriksha repo)

### Out of Scope (Parked for Later AIDLC Runs)

- Tier intelligence / KPI dashboard data (E1-US1 KPI header)

- Mobile / responsive layout

## User Stories

### US-1: List Tiers for a Program

**As a** program manager or API consumer,
**I want to** retrieve all tier configurations for a given program,
**so that** I can view the complete tier hierarchy with all config dimensions.

**Acceptance Criteria:**
- AC-1: `GET /tiers?programId={programId}` returns all tiers for the program, ordered by serialNumber (tier rank).
- AC-2: Each tier in the response includes: id, name, description, color, status (DRAFT/ACTIVE/PENDING_APPROVAL/STOPPED/SNAPSHOT), serialNumber, eligibility config (KPI type, threshold, secondary criteria), validity config (period, type), renewal config (conditions, schedule, duration), downgrade config (target tier, schedule, grace period, validate-on-return-transaction), upgrade config (mode, bonus), nudge/communication config, and linked benefits (summary with per-tier parameter values).
- AC-3: Response is scoped to the requesting org's `orgId` — no cross-tenant data leakage.
- AC-4: Supports filtering by status (e.g., `?status=ACTIVE` to see only live tiers, `?status=DRAFT` to see pending edits).
- AC-5: When a tier has both an ACTIVE version and a DRAFT version (pending edit), the `includeDraftDetails` query param enriches the ACTIVE tier with draft metadata (draft id, version, last modified by/on) — following UnifiedPromotion's `draftDetails` pattern.
- AC-6: Returns empty list (not null, not 404) when program has no tiers.

### US-2: Create a New Tier

**As a** program manager or API consumer,
**I want to** create a new tier configuration for a program,
**so that** I can define a new tier level with all its eligibility, validity, renewal, downgrade, and benefit linkage settings.

**Acceptance Criteria:**
- AC-1: `POST /tiers` accepts a full tier configuration payload including: programId, name (required), description, color, eligibility config (KPI type — required, threshold — required, secondary criteria — optional), validity config (period — required, type — required), downgrade config (target tier ID or "NONE" for base tier — required, schedule, grace period, validate-on-return-transaction toggle), renewal config (conditions, schedule, duration), upgrade config (mode — EAGER/DYNAMIC/LAZY, bonus), nudge/communication config, linked benefit IDs with per-tier parameter overrides.
- AC-2: When maker-checker flag is `true` (default): creates tier with status `DRAFT`. Does NOT go live immediately.
- AC-3: When maker-checker flag is `false`: creates tier with status `ACTIVE` directly.
- AC-4: Assigns a unique immutable `tierId` (UUID) that persists across versions — analogous to UnifiedPromotion's `unifiedPromotionId`.
- AC-5: Assigns `version = 1` on initial creation.
- AC-6: Validates all required fields and returns structured validation errors: `{ "errors": [{ "field": "eligibilityThreshold", "code": "REQUIRED", "message": "Eligibility threshold is required" }] }`. Returns HTTP 422 (not 500) for validation failures.
- AC-7: Validates eligibility threshold is a positive number.
- AC-8: Validates downgrade target references an existing tier in the same program (or is "NONE" for base tier).
- AC-9: Validates that linked benefit IDs reference existing active benefits in the same program.
- AC-10: Validates tier name uniqueness within the program (across all statuses except SNAPSHOT).
- AC-11: Validates KPI type is one of the supported types: CURRENT_POINTS, LIFETIME_POINTS, LIFETIME_PURCHASES, TRACKER_VALUE.
- AC-12: Validates validity type is one of: FIXED_DURATION, REGISTRATION_DATE, FIXED_DATE.
- AC-13: Validates upgrade mode is one of: EAGER, DYNAMIC, LAZY.
- AC-14: Response includes the created tier document with all fields populated, including generated tierId, version, status, and timestamps (createdOn, createdBy).
- AC-15: Request must include orgId (from auth context / header). Tier is scoped to this org.

---

### US-3: Update an Existing Tier

**As a** program manager or API consumer,
**I want to** update an existing tier configuration,
**so that** I can modify any tier setting with full validation and maker-checker support.

**Acceptance Criteria:**
- AC-1: `PUT /tiers/{tierId}` accepts a full or partial tier configuration update payload.
- AC-2: When the existing tier is DRAFT: updates the DRAFT document in-place. No new version is created.
- AC-3: When the existing tier is ACTIVE and maker-checker is `true`: creates a NEW MongoDB document with status `DRAFT`, `version = currentVersion + 1`, `parentId = active document's objectId`, same `tierId`. The ACTIVE version remains unchanged until approval.
- AC-4: When the existing tier is ACTIVE and maker-checker is `false`: updates the ACTIVE document directly.
- AC-5: When a DRAFT already exists for the same tierId (i.e., a pending edit of an ACTIVE tier): updates the existing DRAFT in-place. Does not create a second draft.
- AC-6: All validations from US-2 apply to updates as well.
- AC-7: Cannot update a tier in PENDING_APPROVAL status — returns HTTP 409 Conflict with message "Tier has a pending approval. Approve or reject before making further changes."
- AC-8: Cannot update a tier in STOPPED status — returns HTTP 409 Conflict.
- AC-9: Cannot update a SNAPSHOT — returns HTTP 409 Conflict.
- AC-10: Validates that tier serialNumber/rank changes do not create gaps or conflicts in the program's tier hierarchy.

### US-4: List Benefits for a Program

**As a** program manager or API consumer,
**I want to** retrieve all benefit configurations for a given program,
**so that** I can view the complete benefit catalog with types, categories, linked tiers, and lifecycle state.

**Acceptance Criteria:**
- AC-1: `GET /benefits?programId={programId}` returns all benefits for the program.
- AC-2: Each benefit includes: id (benefitId — immutable UUID), name, type (POINTS_MULTIPLIER, FLAT_POINTS_AWARD, COUPON_ISSUANCE, BADGE_AWARD, FREE_SHIPPING, CUSTOM), category (EARNING, REDEMPTION, COUPON, BADGE, COMMUNICATION, CUSTOM), triggerEvent (TIER_UPGRADE, TIER_RENEWAL, TRANSACTION, BIRTHDAY, MANUAL), description, parameters (type-specific — e.g., multiplier value, points amount, coupon template), linkedTiers (list of tier IDs with per-tier parameter overrides), status (DRAFT/ACTIVE/PENDING_APPROVAL/STOPPED/SNAPSHOT), version, timestamps.
- AC-3: Scoped to requesting org's `orgId`.
- AC-4: Supports filtering by status, type, category, triggerEvent, and linkedTierId.
- AC-5: Supports `includeDraftDetails` and `includeParentDetails` query params for enrichment.
- AC-6: A benefit is scoped to exactly one program. Cannot be shared across programs.
- AC-7: Returns empty list when program has no benefits.

### US-5: Create a New Benefit

**As a** program manager or API consumer,
**I want to** create a new benefit and optionally link it to one or more tiers,
**so that** I can define a standalone benefit with its type, trigger, parameters, and tier associations.

**Acceptance Criteria:**
- AC-1: `POST /benefits` accepts: programId (required), name (required), type (required — one of POINTS_MULTIPLIER, FLAT_POINTS_AWARD, COUPON_ISSUANCE, BADGE_AWARD, FREE_SHIPPING, CUSTOM), category (required), triggerEvent (required), description, parameters (type-specific JSON — e.g., `{ "multiplier": 2.0 }` for POINTS_MULTIPLIER, `{ "points": 500 }` for FLAT_POINTS_AWARD), linkedTiers (optional — array of `{ "tierId": "...", "parameters": { ... } }` with per-tier parameter overrides).
- AC-2: When maker-checker is `true`: creates with status DRAFT. When `false`: creates with status ACTIVE.
- AC-3: Assigns immutable `benefitId` (UUID) and `version = 1`.
- AC-4: Validates name uniqueness within the program.
- AC-5: Validates linkedTier IDs reference existing tiers in the same program.
- AC-6: Validates type-specific parameters are present and valid (e.g., POINTS_MULTIPLIER requires `multiplier` > 0; FLAT_POINTS_AWARD requires `points` > 0).
- AC-7: Returns structured validation errors on failure (HTTP 422).
- AC-8: Benefit is scoped to one program (programId). Cannot be linked to tiers in other programs.
- AC-9: Response includes the full created benefit document with generated benefitId, version, status, timestamps.

### US-6: Update an Existing Benefit

**As a** program manager or API consumer,
**I want to** update an existing benefit configuration,
**so that** I can modify benefit settings, parameters, or tier linkage with maker-checker support.

**Acceptance Criteria:**
- AC-1: `PUT /benefits/{benefitId}` accepts a full or partial update payload.
- AC-2: Follows the same versioning/draft pattern as tier updates (US-3 AC-2 through AC-9): DRAFT updates in-place, ACTIVE creates a new DRAFT version with parentId linkage, existing DRAFT for same benefitId is updated in-place.
- AC-3: Validates linkedTier IDs reference existing tiers in the same program.
- AC-4: Validates type-specific parameters (same as US-5).
- AC-5: Cannot update PENDING_APPROVAL, STOPPED, or SNAPSHOT benefits — returns HTTP 409.

---

### US-7: Submit Tier/Benefit for Approval

**As a** program manager or API consumer,
**I want to** submit a DRAFT tier or benefit for approval,
**so that** it enters the maker-checker review queue.

**Acceptance Criteria:**
- AC-1: `PUT /tiers/{tierId}/status` with body `{ "status": "PENDING_APPROVAL", "reason": "optional comment" }` transitions a DRAFT tier to PENDING_APPROVAL.
- AC-2: `PUT /benefits/{benefitId}/status` with body `{ "status": "PENDING_APPROVAL", "reason": "optional comment" }` transitions a DRAFT benefit to PENDING_APPROVAL.
- AC-3: Only valid transition: DRAFT → PENDING_APPROVAL. Any other source status returns HTTP 409 with the invalid transition error.
- AC-4: Concurrency control: uses distributed lock on `lock_tier_status_{orgId}_{tierId}` (TTL 300s) to prevent race conditions — following UnifiedPromotion's `@Lockable` pattern.
- AC-5: Stores the submission reason/comment on the document.

---

### US-8: Approve a Tier/Benefit

**As an** approver or API consumer,
**I want to** approve a pending tier or benefit configuration change,
**so that** it goes live in the program.

**Acceptance Criteria:**
- AC-1: `POST /tiers/{tierId}/review` with body `{ "approvalStatus": "APPROVE", "comment": "optional, max 150 chars" }` approves a PENDING_APPROVAL tier.
- AC-2: **New tier approval** (parentId is null): status flips from PENDING_APPROVAL → ACTIVE. This is a net-new tier going live.
- AC-3: **Edit approval** (parentId is not null): the ACTIVE parent tier transitions to SNAPSHOT. The approved draft transitions to ACTIVE (inheriting the parent's previous status). This preserves the version history.
- AC-4: Stores the approval comment on the document.
- AC-5: `POST /benefits/{benefitId}/review` follows the same pattern for benefits.
- AC-6: Only valid source status: PENDING_APPROVAL. Returns HTTP 409 for any other status.
- AC-7: Concurrency control via distributed lock (same as US-7).
- AC-8: After approval, the tier/benefit is queryable with its new ACTIVE status in GET endpoints.

---

### US-9: Reject a Tier/Benefit

**As an** approver or API consumer,
**I want to** reject a pending tier or benefit configuration change,
**so that** it returns to DRAFT for further editing.

**Acceptance Criteria:**
- AC-1: `POST /tiers/{tierId}/review` with body `{ "approvalStatus": "REJECT", "comment": "mandatory, max 150 chars" }` rejects a PENDING_APPROVAL tier.
- AC-2: Status transitions from PENDING_APPROVAL → DRAFT. The submitter can edit and resubmit.
- AC-3: Rejection comment is mandatory (unlike approval where it's optional). Returns HTTP 422 if comment is missing.
- AC-4: `POST /benefits/{benefitId}/review` follows the same pattern for benefits.
- AC-5: The rejected tier/benefit retains its parentId and version — it can be resubmitted after edits.

---

### US-10: Stop/Pause a Tier or Benefit

**As a** program manager or API consumer,
**I want to** stop or pause an active tier or benefit,
**so that** it is deactivated without being deleted.

**Acceptance Criteria:**
- AC-1: `PUT /tiers/{tierId}/status` with `{ "status": "STOPPED" }` transitions ACTIVE → STOPPED.
- AC-2: `PUT /tiers/{tierId}/status` with `{ "status": "PAUSED" }` transitions ACTIVE → PAUSED.
- AC-3: `PUT /tiers/{tierId}/status` with `{ "status": "ACTIVE" }` (resume) transitions PAUSED → ACTIVE.
- AC-4: STOPPED is terminal — no further transitions allowed. Returns HTTP 409 if attempting to change from STOPPED.
- AC-5: Same status transitions apply to benefits via `PUT /benefits/{benefitId}/status`.
- AC-6: Valid transitions enforced by a status transition validator (following `StatusTransitionValidator` pattern):
  - DRAFT → PENDING_APPROVAL
  - PENDING_APPROVAL → ACTIVE (approve), DRAFT (reject)
  - ACTIVE → PAUSED, STOPPED
  - PAUSED → ACTIVE (resume), STOPPED
  - STOPPED → (terminal, no transitions)
  - SNAPSHOT → (terminal, no transitions)

### US-11: List Pending Approvals

**As an** approver or API consumer,
**I want to** see all pending tier and benefit changes awaiting approval,
**so that** I can review and act on them.

**Acceptance Criteria:**
- AC-1: `GET /tiers/approvals?programId={programId}` returns all tiers in PENDING_APPROVAL status for the program.
- AC-2: `GET /benefits/approvals?programId={programId}` returns all benefits in PENDING_APPROVAL status for the program.
- AC-3: Each pending item includes: the full config document, the diff against the parent (for edits — old value vs new value), who submitted it (createdBy), when (createdOn), and the submission reason.
- AC-4: For edit approvals (parentId not null): includes `parentDetails` — the current ACTIVE version's key fields for comparison.
- AC-5: Scoped to requesting org's `orgId`.

## Constraints

- **MongoDB storage**: All tier and benefit configuration documents are stored in MongoDB (not MySQL strategy tables). This creates a new data path alongside the legacy MySQL strategy tables. The legacy tables remain the source of truth for EMF evaluation until a migration strategy is defined.
- **No dual writes to legacy**: This iteration creates the new MongoDB-based API layer only. Synchronization between the new MongoDB tier configs and legacy MySQL strategy tables (consumed by EMF evaluation engine) is out of scope and will be addressed in a subsequent iteration.
- **Backend only**: No frontend changes. The API must be self-sufficient — all validation in the API, not in UI form logic.
- **emf-parent repo**: All code lives in kalpavriksha/emf-parent.
- **Multi-tenancy**: Every endpoint must enforce orgId scoping. OrgId is extracted from the auth context (JWT/header) at the service boundary.
- **No notifications**: Approval/rejection does not trigger email or in-platform notifications in this iteration.
- **No deletion**: Tiers and benefits cannot be deleted — only STOPPED (soft delete). This preserves referential integrity with historical data.
- **Maker-checker always on** (for now): Config flag defaults to `true`. The bypass path (flag = `false`, direct to ACTIVE) should be implemented but is not the primary flow.
- **Benefit scoped to one program**: A benefit can only belong to one program. Cross-program benefit sharing is out of scope.
- **Tier ordering**: Tiers have a serialNumber (rank) that defines the hierarchy. The base tier has the lowest serialNumber. New tiers are added at a specific position. Tiers cannot be reordered arbitrarily without validating downgrade target references.
- **No tier deletion**: "You cannot delete an existing tier" — consistent with current platform constraint. Only STOP.
- **Terminology bridge**: API uses "tier" externally (field names, endpoint paths). Internal code may continue to use "slab" where it interfaces with existing EMF components.

## Assumptions

- The UnifiedPromotion maker-checker pattern in intouch-api-v3 is the approved reference implementation. Our tier/benefit maker-checker will follow the same DRAFT → PENDING_APPROVAL → ACTIVE lifecycle, parentId versioning, and distributed locking approach.
- MongoDB is already available in the emf-parent infrastructure (confirmed: `springdata-mongodb` dependency exists, `moonknight` collections are in use for OrgConfig and PromotionConfig).
- The config flag for maker-checker toggle will be a simple boolean property (hardcoded `true`) that can later be moved to a DB-backed program-level setting.
- Authorization (who can approve, who can submit) is handled at the UI/gateway layer, not in the backend API. The API provides the mechanics (create draft, approve, reject) without enforcing role checks.
- The new benefit entity is a **net-new MongoDB document** — not a facade over V3 Promotions. This means benefits created through the new API are independent of the existing PromotionConfig/UnifiedPromotion system.
- Integration between new benefit entities and the EMF evaluation engine (so that benefits actually get awarded to members) is a subsequent concern — this iteration establishes the configuration data model and API surface.
- "Validate downgrade condition for return transaction" is included as a boolean toggle field on the tier downgrade configuration.
- Renewal condition in the API maps to the existing retention metrics concept: `retentionAmount`, `retentionPoints`, `retentionTracker`, `retentionVisits`.
- The API will use ISO-8601 timestamps with timezone (UTC) per guardrail G-01.6.
- Benefit trigger events (TIER_UPGRADE, TIER_RENEWAL, TRANSACTION, BIRTHDAY, MANUAL) are stored as enum values on the benefit document. Whether all trigger events are wired to the EMF evaluation pipeline is a subsequent integration concern.
