## Alignment with Current Product

### Confirmed Alignments

- **Tier = "Slab" internally** — The BRD consistently uses "tier" and "benefit". The codebase uses "slab" for tiers throughout: `ProgramSlab`, `SlabUpgradeService`, `SlabDowngradeService`, `SlabChangeSource`, `TierRulesetNamePrefix`. Both refer to the same concept. No conflict — just a terminology layer to bridge.
- **Tier lifecycle events exist** — `SLAB_UPGRADE`, `SLAB_DOWNGRADE`, `SLAB_RENEWAL` are first-class instruction mnemonics in `EMFInstruction.Mnemonic`. The BRD's upgrade/downgrade/renewal model aligns with existing platform behaviour.
- **Programs are multi-slab** — `Program` entity has a `List<ProgramSlab> programSlabs` relationship (`program_slabs` table). The BRD's concept of "tiers within a program" is correctly modelled.
- **Tier events publish to Kafka** — `TierUpgradeHelper` and `TierRenewedHelper` both call `EventProducer.getInstance().sendAsync(...)` to publish tier events downstream. The BRD's mention of "communications triggered by tier events" is consistent.
- **SlabUpgradeMode exists** — The `SlabUpgradeMode` enum (`EAGER`, `DYNAMIC`, `LAZY`) documents that tier upgrade timing relative to event processing is already configurable. The BRD's "upgrade schedule" concept maps to this.
- **SlabChangeSource exists** — `SlabChangeSource` enum includes `RULE, STRATEGY, GROUP_SYNC, MERGE, IMPORT, GROUP_LEAVE, PARTNER_PROGRAM, MANUAL_SLAB_ADJUSTMENT`. This shows the platform already tracks the reason for a tier change — relevant to the BRD's Change Log requirement.
- **Benefits exist as a tracked entity** — `BenefitsAwardedStats` entity (`benefits_awarded_stats` table) tracks awarded benefits with `BenefitType` (REWARDS, COUPONS, BADGES, TIER_UPGRADE, TIER_DOWNGRADE, TIER_RENEWAL, ENROL, OPTIN, PARTNER_PROGRAM, etc.) and `ContextType` (PROGRAM, PROMOTION). The platform already records benefit awards.
- **Audit trail for tier changes exists** — `SlabUpgradeAuditLogService` and `SlabDowngradeAuditLogService` exist in `pointsengine-emf`. These use `AuditTrail`, `AuditLogTrailDetailsDto`, `TierConfiguration`, and `AuditTrailDiffDto` structures. The BRD's Change Log requirement (E1-US5) has existing infra to build on.
- **Promotions are the current benefit delivery mechanism** — `PromotionConfig` (MongoDB, `moonknight` collection) and `UnifiedPromotion` (`unified_promotions` collection) confirm benefits are currently delivered via the V3 Promotions engine. The BRD explicitly acknowledges this is the problem it is solving.
- **Promotion maker-checker exists in UnifiedPromotion** — `UnifiedPromotion` has `comments` (review comment during approval/rejection), `parentId` (points to original when an ACTIVE promotion is updated as a new draft), and `version` fields. The approval pattern (create draft → review → approve/publish) exists in the promotions domain already.
- **OrgId-scoped multi-tenancy is established** — Every core entity (`BenefitsAwardedStats`, `Program`, `ProgramSlab`, `PromotionConfig`, `UnifiedPromotion`) carries `orgId` as part of a composite primary key or as a field. Multi-tenancy is enforced at the data model level via org-scoping.
- **Simulation infrastructure exists in EMF** — `EMFThriftServiceImpl.simulateRulesBasedOnDateRange()` is an existing method that replays bills within a date range. It is Thrift-based and processes up to 5,000 bills. The BRD's simulation mode has a foundation to build on, though the existing API is batch/historical — not the member-distribution forecast the BRD needs.
- **MPL (Multi-Program Loyalty) is already handled** — The `processSlabInstructions` logic in `EMFThriftServiceImpl` removes duplicate slab instructions from the default program in MPL scenarios. The BRD does not mention MPL, but it is an active concern in the platform.

|---|---|---|---|---|

| CC-4 | `[Infra]` | **aiRa Context Layer data freshness and caching** | The `GET /program/{programId}/context` response includes member distribution counts. For large orgs, this is an expensive aggregate query. The BRD does not specify: caching strategy, TTL, or what happens when the context is stale during an aiRa session that spans multiple turns. | open |
| CC-5 | `[Backend]` | **Event-driven side effects of tier config changes** | Changing a tier threshold (e.g., lowering Gold from $500 to $400) can trigger immediate member re-evaluations at the next `SLAB_RENEWAL` or batch evaluation cycle. The BRD describes an "impact preview" but does not specify: when does the re-evaluation actually happen (immediately on publish, or at next batch cycle)? Does an immediate re-evaluation risk overloading the EMF batch pipeline? | open |
| CC-6 | `[Cross-team]` | **Backward compatibility of tier config changes** | Existing promotions in `PromotionConfig`/`UnifiedPromotion` may have rules that reference specific slab serial numbers (e.g., `slabNumber == 3`). If a tier is deleted or reordered via the new API, these serial number references break silently. The BRD does not address cascade validation — what the platform does when a tier referenced by a promotion is deleted. | open |

## The 6 BRD Open Questions — Analysis

The following analysis addresses the 6 open questions from Section 12 of the BRD directly:

| BRD Q# | Question | ProductEx Finding | Risk |
|---|---|---|---|
| Q1 | Does the program context API exist? Latency at p99? | **Does not exist** in the visible codebase. `OrgConfiguration` is the closest analogue but covers ruleset config only — not member distribution, benefit catalog, or change history. Building this requires aggregating data from at least: `ProgramSlab` (tier structure), customer enrollment tables (member distribution), MongoDB promotion stores (benefit catalog), and audit log tables (change history). This is a **net-new data aggregation service** with significant build cost. | **Blocking for aiRa (E3).** Without this API, aiRa is a generic chatbot, not a loyalty-aware assistant. |
| Q2 | Simulation — real-time or queued? Acceptable SLA? | Existing simulation in `EMFThriftServiceImpl.simulateRulesBasedOnDateRange` is synchronous over Thrift, limited to 5,000 bills, and is a rule-replay engine — not a member distribution projector. The BRD's simulation (member count before/after threshold change) is a different computation: a COUNT query on enrollment tables filtered by proposed threshold, which should be faster. However, for multi-million-member programs, even a COUNT with index may take seconds. **Real-time is plausible for sub-million-member orgs; queued is safer for enterprise.** | SLA must be defined before the simulation API and UX can be designed. |
| Q3 | Maker-checker per-role or per-program? Who configures approvers? | **Not implemented for tier config** in any visible codebase component. Partial pattern exists in `UnifiedPromotion` (parentId/version/comments) for promotions. For tiers, there is no approval mechanism at all. The answer to this question determines: the data model for pending changes, the notification routing logic, and whether approver assignment is a Garuda admin-level setting or a program-level setting. | **Blocking for E1-US4.** Cannot build approvals without this decision. |
| Q4 | Can a benefit be linked to multiple programs, or always scoped to one? | All existing benefit/promotion entities (`PromotionMetadata.programId`, `PromotionConfig.contextId`) suggest **program-scoped** today. `BenefitsAwardedStats` has `contextIdentifier` (int) and `contextType` (PROGRAM or PROMOTION) — no cross-program linkage. Building multi-program benefit sharing would require a new many-to-many linking entity. | Impacts E2-US2 data model. If cross-program is needed, it is a significant backend effort. |
| Q5 | Should aiRa handle multi-turn disambiguation, or single-turn? | **Not determinable from codebase** — aiRa backend not present in repo. From a product perspective: single-turn is simpler to build but produces worse UX for complex configs (e.g., creating a new tier requires >5 parameters). Multi-turn produces better UX but requires session state management. The BRD's example conversations (Section 5, 8.2) clearly show multi-turn exchanges — the BRD implicitly assumes multi-turn. | This decision affects aiRa backend architecture. If multi-turn is confirmed, session storage (Redis or similar) for conversation state is an Infra requirement. |
| Q6 | Should "validate downgrade condition for return transaction" toggle be surfaced or deprecated? | The `RETURN` mnemonic is active in `EMFInstruction.Mnemonic`. `EMFThriftServiceImpl` and `TxProfileImpl` both handle return transactions. The toggle likely maps to whether a return transaction triggers a `SLAB_DOWNGRADE` evaluation. **Deprecating without a replacement migration plan risks breaking orgs** that intentionally suppress tier downgrade on returns. Surfacing it requires UX copy. | Surface it with clear copy in Phase 1; schedule deprecation for Phase 2 with a migration plan. |

### Recommendation

**Pause for product team input on 5 blocking gaps before proceeding to Architect phase:**

1. **PB-1**: Decide whether "Benefits as a Product" is a façade over V3 Promotions or a new entity. This is the foundational data model decision for all of Epic 2.
2. **D-1**: Formally decide on Option A (Hybrid) for Phase 1. BRD text suggests this but does not state it as a decision.
3. **BT-3 / BT-4**: Determine what backend service owns tier config writes and what the data model for "pending tier change" looks like. Architect cannot scope Epic 4 (APIs) without this.
4. **CC-1**: Define the multi-tenancy contract for all new APIs (how orgId is passed, validated, enforced).
5. **Q1 (BT-1)**: Establish whether the program context API is a new build or an existing endpoint — and if new, accept it as a prerequisite to E3 (aiRa) being buildable.

The remaining questions (25+) should be addressed in the BA Q&A phase and can be resolved iteratively. They do not block the Architect from starting scoping, provided the 5 blocking gaps above are resolved.
