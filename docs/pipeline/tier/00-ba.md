## Current Behaviour

### How Tiers Work Today

Capillary's loyalty platform uses "slabs" (internal term for tiers) to define membership levels within a loyalty program. The current behaviour, documented at docs.capillarytech.com and verified against the codebase:

**Data Model:**
- `ProgramSlab` entity (table: `program_slabs`) stores: id, orgId, programId, serialNumber, name, description, metadata (JSON blob), createdOn. It has NO status field, NO color field, and NO inline eligibility configuration.
- `PartnerProgramSlab` extends this for multi-loyalty partner programs.
- Tier configuration (eligibility, upgrade thresholds, downgrade rules, renewal conditions) is stored as Gson-serialized JSON in strategy configuration objects, NOT in the ProgramSlab entity itself.
- `TierConfiguration` DTO aggregates: isActive flag, per-slab downgrade configs (TierDowngradeSlabConfig[]), alert configs, renewal confirmation, threshold values, and tracker references.
- `SlabMetaData` stores colorCode and is parsed from the metadata JSON.

**Eligibility & Upgrade:**
- Four eligibility criteria types: Current Points, Lifetime Points, Lifetime Purchases, Tracker Value. The same criteria type applies to ALL tiers in a program.
- Three upgrade types: Issue Points Then Upgrade, Upgrade Then Issue Points, Issue-Upgrade-Then-Issue-Remaining (controls when tier evaluation happens during transaction processing).
- `SlabUpgradeMode` enum: EAGER (before awarding), DYNAMIC (during awarding), LAZY (after awarding).
- Strategies: `SlabUpgradeStrategy` interface with `SlabUpgradeStrategyImpl` and `ThresholdBasedSlabUpgradeStrategyImpl` implementations.

**Downgrade & Renewal:**
- Downgrade targets: SINGLE (one tier below), THRESHOLD (to matching tier), LOWEST (base tier).
- Downgrade conditions: purchase amount, visit count, points, tracker values, with AND/OR logic.
- Renewal condition types: Any (N-1), All, Custom (AND/OR with parentheses).
- PEB (Points Engine Backend) handles bulk tier downgrade/reassessment via `TierDowngradeBatchServiceImpl`, `TierReassessmentServiceImpl`, and various `GapToUpgradeCalculator` implementations.

**APIs:**
- NO REST API endpoints exist for tier CRUD in intouch-api-v3 (verified: zero controllers/facades reference tiers/slabs).
- All current tier operations go through Thrift (EMFThriftServiceImpl) or internal service calls.
- Tier creation uses `PointsEngineRuleService.createSlabAndUpdateStrategies()` which creates the ProgramSlab record AND updates associated strategy configurations.

**Maker-Checker (Approval Framework):**
- Generic approval framework exists in makechecker package for unified promotions (UnifiedPromotionFacade, ApprovalStatus, PromotionReviewRequest).
- No approval flow exists yet for tier configuration.
- The promotion approval pattern uses: MongoDB draft storage (`@Document(collection = "unified_promotions")`), `PromotionStatus` lifecycle (DRAFT -> PENDING_APPROVAL -> ACTIVE -> PAUSED -> STOPPED), `EntityOrchestrator` with ApprovableEntityHandler pattern to sync MongoDB -> SQL on approval.

**UI:**
- Current TiersView.js in Garuda/cap-loyalty-ui renders a flat list of tier cards from a single API. No comparison, no configuration entry, no KPI data, no status visibility, no action surface.

### Constraints from Current System
- Tiers must be created in sequential order (cannot insert between existing tiers).
- Existing tiers cannot be deleted (per docs).
- The `serialNumber` ordering is used by upgrade/downgrade logic and must remain stable.
- `customer_enrollment` table tracks each member's `current_slab_id` -- any change to tier structure must preserve referential integrity.

## Problem Statement

Loyalty program managers cannot efficiently manage tier configurations because the experience is fragmented across six screens, tier logic is invisible, and there are no APIs for programmatic tier management. Creating a single tier takes ~45 minutes. There is no comparison view, no lifecycle management, no approval workflow, and no way to see impact before committing changes.

---

## Goals

1. **Tier CRUD APIs**: Self-reliant REST APIs for listing, creating, editing, and deleting tiers -- with validation in the API, not the UI.
2. **Comparison Matrix Data**: A single API call returns all tiers in a program with full configuration across all dimensions (eligibility, renewal, downgrade, benefits linkage).
3. **Generic Maker-Checker Framework**: A reusable approval workflow that tiers plug into first, with benefits, subscriptions, and other entities plugging in later.
4. **Tier Lifecycle Management**: Status-based lifecycle (Draft, Pending Approval, Active, Deleted) with versioned edits and rollback capability. No PAUSED or STOPPED states. Tier retirement (stopping ACTIVE tiers) is deferred to a future epic.
5. **Extensible Architecture**: Design supports future change log (E1-US5), simulation mode (E1-US6), tier retirement, and benefits module (E2) without rework.
6. **Tier Ordering Immutability**: Tier reordering is explicitly not supported. serialNumber is auto-assigned and immutable.
7. **Unified Read Surface (Rework #5)**: The new `/v3/tiers` API returns **all** tiers of a program — both legacy SQL-origin tiers and new-UI-origin tiers. Legacy tiers surface through a read-only SQL→DTO converter; no bootstrap migration into Mongo.
8. **Dual Write Paths, Single MC Scope (Rework #5)**: Maker-checker applies **if-and-only-if** the write originated from the new UI. Old UI writes continue via the legacy direct-SQL path with no MC. New UI writes always go through Mongo DRAFT → MC → Thrift → SQL.
9. **Drift Detection Before Approval (Rework #5)**: Each Mongo DRAFT captures a basis SQL snapshot at draft-creation time. At approval time, the current SQL state is compared against the basis; any drift blocks the approval with a clear user-facing message.
10. **SQL as Single Source of Truth for LIVE (Rework #5)**: The live state of every tier — legacy or new-UI-origin — is always read from SQL `program_slabs`. Mongo holds in-flight DRAFT/PENDING_APPROVAL and audit-only SNAPSHOT docs. SNAPSHOTs are write-once-at-approval and never reconciled with later legacy edits.

## Scope

### In Scope
- **E1-US1**: Tier Listing with Comparison Matrix (API layer) — **unified over SQL + Mongo** per Rework #5
- **E1-US2**: Tier Creation (API + MongoDB draft + maker-checker) — new UI path only; old UI retains legacy direct-SQL create
- **E1-US3**: Tier Editing (versioned edits, inline + structural) — new UI path only; old UI retains legacy direct-SQL edit
- **Tier Deletion (DRAFT only)**: Soft-delete of DRAFT tiers to DELETED status. ACTIVE tiers cannot be deleted or retired in this pipeline run.
- **Generic Maker-Checker Framework**: Shared module (`makechecker/` package — `ApprovableEntity`, `ApprovableEntityHandler<T>`, `MakerCheckerService<T>`, SAGA pattern)
- **Tier Status Lifecycle**: DRAFT -> PENDING_APPROVAL -> ACTIVE. DRAFT -> DELETED (terminal). PENDING_APPROVAL -> SNAPSHOT (on approval, audit only). No PAUSED or STOPPED states.
- **Tier Ordering**: serialNumber is immutable. Tier reordering is explicitly not supported.
- **KPI Header**: Total Tiers, Active Tiers, Pending Approval, Total Members (cached)
- **Unified Read Surface (Rework #5)**: `/v3/tiers` GETs return both legacy SQL-origin and new-UI-origin tiers. LIVE state always sourced from SQL `program_slabs` via read-only SQL→DTO converter.
- **Envelope Response Shape (Rework #5)**: Tier detail and list endpoints return `{ live: ..., pendingDraft: ... | null }` per tier in a single round-trip.
- **Dual Write Paths (Rework #5)**:
  - Old UI write → legacy direct-SQL write (unchanged legacy flow, no MC, no Mongo).
  - New UI write → Mongo DRAFT (snapshot from current SQL if editing a legacy tier) → MC → Thrift → SQL.
- **Drift Detection at Approval (Rework #5)**: Every DRAFT captures `meta.basisSqlSnapshot` at creation; approval flow re-reads current SQL and blocks on any drift.
- **SNAPSHOT Audit-Only (Rework #5)**: Post-approval SNAPSHOT docs are the audit record. Not reconciled with later legacy edits. History view surfaces a caveat when last SNAPSHOT's `approvedAt` < current SQL `updated_at`.
- **Single Active DRAFT per Tier (Rework #5)**: At most one DRAFT or PENDING_APPROVAL per existing tier. Enforced at app level (friendly error) + Mongo partial unique index on `(orgId, programId, slabId)` where `status IN [DRAFT, PENDING_APPROVAL]`.
- **Name Collision Defense (Rework #5)**: Three-layer — app-level check at DRAFT creation, re-check at approval time, and SQL `UNIQUE (program_id, name)` as final backstop.
- **Schema Cleanup (Rework #5)**:
  - Drop `nudges` from `UnifiedTierConfig` (standalone `Nudges` entity untouched).
  - Drop `benefitIds` — tiers have no knowledge of benefits.
  - Drop `metadata.updatedViaNewUI` — not required.
  - Drop `basicDetails.startDate`/`endDate` — UI Duration column removed.
  - Hoist `basicDetails` and `metadata` fields to root of `UnifiedTierConfig` (eliminate both wrappers).
  - Rename `unifiedTierId` → `tierUniqueId` (pure rename; same format `"ut-977-004"`).
  - Rename `metadata.sqlSlabId` → `slabId` (= `program_slabs.id`, root-level after hoist).
- **SQL Audit Columns (Rework #5)**: `program_slabs` gains `updatedBy`, `approvedBy`, `approvedAt`. `createdBy` is explicitly NOT added. Legacy writes set only `updatedBy`; MC-push writes set all three.

### Out of Scope
- **Tier Retirement**: Stopping/deactivating ACTIVE tiers is out of scope. No PAUSED or STOPPED states. Future epic.
- **Tier Reordering**: serialNumber is immutable and auto-assigned. No API to reorder tiers.
- **Bootstrap Migration of Legacy Tiers (Rework #5)**: Legacy SQL-only tiers are NOT copied into Mongo. They surface via the read-only SQL→DTO converter on GETs.
- **Reconciling Mongo SNAPSHOTs with Legacy Edits (Rework #5)**: SNAPSHOTs are write-once-at-approval audit records. Legacy edits after approval do not update the SNAPSHOT. UI reconciles visually via a caveat label.
- **`createdBy` on `program_slabs` (Rework #5)**: Not added — only `updatedBy`, `approvedBy`, `approvedAt`.
- **MC for Old-UI Writes (Rework #5)**: Old UI writes bypass MC entirely — legacy direct-SQL behaviour is preserved.
- **E1-US4**: Maker-Checker Approval Queue UI (the framework is built, but the dedicated approval queue view is deferred)
- **E1-US5**: Change Log (architecture supports it, not built)
- **E1-US6**: Simulation Mode (architecture supports it, not built)
- **E2**: Benefits as a Product (separate epic)
- **E3**: aiRa Configuration Layer (separate epic)
- **UI implementation**: This pipeline builds backend APIs only
- **Real-time member count queries**: Using cached counts instead

## User Stories

### US-1: Tier Listing with Comparison Matrix
**As** a loyalty program manager,
**I want** to retrieve all tiers in my program with full configuration displayed in a comparable format,
**So that** I can understand the tier structure, identify gaps, and spot inconsistencies from a single API call.

**Acceptance Criteria:**
- [ ] `GET /v3/tiers?programId={id}` returns all tiers for a program — **both legacy SQL-origin and new-UI-origin** (Rework #5)
- [ ] Response shape is an **envelope per tier**: `{ live: {...}, pendingDraft: {...} | null }` (Rework #5)
- [ ] `live` is always sourced from SQL `program_slabs` via a read-only SQL→DTO converter — single source of truth for live state (Rework #5)
- [ ] `pendingDraft` is sourced from Mongo `UnifiedTierConfig` filtered by `status IN [DRAFT, PENDING_APPROVAL]`, joined in-memory by `slabId`; `null` if none
- [ ] Response includes per-tier: basic details (name, description, color, status), eligibility criteria (type, threshold, activities with AND/OR logic), membership duration, upgrade schedule, notifications (in-tier notification config only)
- [ ] Response includes per-tier: renewal config (`criteriaType` — R5-R4 B1a: always `"Same as eligibility"` today; `expressionRelation` and `conditions` are null/empty until B2/B3). `schedule` and `notifications` are out of scope (dropped in Rework #5). `renewal` is never null on the wire — if no DRAFT exists, the SQL-sourced LIVE view synthesizes the same default.
- [ ] Response includes per-tier: downgrade config (downgrade-to tier, schedule, expiry reminders)
- [ ] Response does NOT include `benefitIds` or benefit details (Rework #5 — tiers have no knowledge of benefits; Benefits epic owns the link)
- [ ] Response does NOT include `nudges` field (Rework #5 — standalone Nudges entity is separate)
- [ ] Response includes KPI header: total tiers, active tiers, scheduled tiers, total members (cached)
- [ ] Response includes per-tier member count (cached, refreshed every 5-15 minutes)
- [ ] Response includes status badge per tier: Active, Draft, Pending Approval
- [ ] Filterable by status (active, draft, pending_approval). Deleted tiers excluded by default (unless `?includeDeleted=true`)
- [ ] Tiers ordered by serialNumber (ascending) — reordering not supported
- [ ] Two DB hits per list page maximum: one SQL query for LIVE rows, one Mongo query for DRAFT/PENDING docs. No N+1.
- [ ] Mongo index `(orgId, programId, status)` covers the DRAFT/PENDING query for list
- [ ] For the **old UI's legacy listing endpoint**, behaviour is unchanged — it reads SQL only, is blind to Mongo, and returns the classic response shape (Rework #5)

### US-2: Tier Creation
**As** a loyalty program manager,
**I want** to create a new tier with full configuration,
**So that** I can extend my program's tier hierarchy.

**Acceptance Criteria:**
- [ ] `POST /v3/tiers` (new UI) creates a new tier via the Mongo DRAFT → MC → Thrift → SQL path (Rework #5)
- [ ] On DRAFT creation for a **new** tier, `slabId` is `null`; assigned by SQL on approval (Thrift INSERT returns the generated id)
- [ ] Required fields: name, programId, eligibilityCriteriaType, eligibilityThreshold
- [ ] Optional fields: description, color, upgradeSchedule, membershipDuration, renewalConfig (R5-R4 B1a — accepts only `criteriaType = "Same as eligibility"`; any other value is rejected 400; a null or omitted `renewal` is auto-filled to the B1a default by `TierRenewalNormalizer` before persistence), downgradeConfig (no `nudges`, no `benefitIds` per Rework #5)
- [ ] Validation in API (at DRAFT creation — Layer 1 of 3 per Rework #5):
  - Name uniqueness across (SQL `program_slabs` for that program) ∪ (active Mongo DRAFTs for that program); reject early on collision
  - Threshold > previous tier's threshold
  - `parentId` (if set) must reference an existing LIVE tier's `slabId` in the same program; a DRAFT cannot be a parent
- [ ] serialNumber auto-assigned = max(existing serialNumbers) + 1 (on approval, from SQL)
- [ ] Name re-check at approval time (Layer 2 per Rework #5): if collision appeared since DRAFT creation (e.g., via old UI), approval is blocked with: *"Name '<X>' no longer available — conflicts with existing tier. Please rename or cancel this DRAFT."*
- [ ] SQL `UNIQUE (program_id, name)` constraint is the final backstop (Layer 3 per Rework #5)
- [ ] If maker-checker enabled (per org+program+entityType): saves to MongoDB as DRAFT, returns tier with status=DRAFT
- [ ] If maker-checker disabled: saves to MongoDB AND syncs to SQL immediately, returns tier with status=ACTIVE (rare path in practice; both paths use the same SAGA)
- [ ] **Old UI continues using the legacy create endpoint** — direct SQL INSERT, no MC, no Mongo write (Rework #5)
- [ ] Returns structured validation error object (field-level, human-readable) on invalid input — not a 500
- [ ] MongoDB document uses hoisted root-level fields after Rework #5 (no `basicDetails` / `metadata` wrappers); `tierUniqueId` (ex `unifiedTierId`) is auto-generated
- [ ] DRAFT captures `meta.basisSqlSnapshot = null` for new-tier creates (no basis SQL row yet) (Rework #5)
- [ ] At DRAFT creation, app-level check enforces "single active DRAFT per tier" (Rework #5): for new tiers, multiple DRAFTs with `slabId=null` may coexist (each gets its own slab on approval); for edits of existing LIVE tiers, any pre-existing DRAFT or PENDING_APPROVAL for that `slabId` blocks a new DRAFT

### US-3: Tier Editing
**As** a loyalty program manager,
**I want** to edit an existing tier's configuration with full impact visibility,
**So that** I can adjust tier logic without breaking the live program.

**Acceptance Criteria:**
- [ ] `PUT /v3/tiers/{tierId}` (new UI) updates a tier via the dual-storage MC path (Rework #5)
- [ ] Editing a DRAFT tier (new-UI-origin): updates the Mongo document in place
- [ ] Editing a LIVE tier (either legacy SQL-origin OR previously-approved new-UI-origin) via new UI: creates a **new Mongo DRAFT document**; the `live` SQL row stays live and visible to all UIs. `slabId` on the DRAFT = `program_slabs.id` of the LIVE tier.
- [ ] On DRAFT creation for a LIVE-tier edit: the DRAFT captures `meta.basisSqlSnapshot` = snapshot of current SQL state of the tier (fields governed by MC). Used for drift detection at approval.
- [ ] Editing a PENDING_APPROVAL tier: updates the pending document in place (or rejects and re-submits)
- [ ] All editable fields from creation are editable (name, description, color, eligibility, renewal, downgrade). Renewal edits follow the same R5-R4 B1a contract (accept-only `"Same as eligibility"`; null auto-filled pre-save). `nudges` and `benefitIds` are NOT in scope (Rework #5).
- [ ] serialNumber is NOT editable (preserves tier ordering)
- [ ] Returns validation errors on invalid input
- [ ] Single active DRAFT/PENDING_APPROVAL per tier enforced (Rework #5):
  - App-level: before insert, query `{orgId, programId, slabId, status IN [DRAFT, PENDING_APPROVAL]}`; if one exists, reject with: *"Tier '<name>' already has a pending change (status: <DRAFT|PENDING_APPROVAL>) created by <user> on <date>. Please review or cancel it before making new changes."*
  - DB-level: Mongo partial unique index on `(orgId, programId, slabId)` where `status IN [DRAFT, PENDING_APPROVAL]` catches races.
- [ ] On approval of a LIVE-tier edit: the Mongo DRAFT transitions PENDING_APPROVAL → SNAPSHOT (audit); SQL `program_slabs` row is updated in place with the approved fields, `updatedBy`/`approvedBy`/`approvedAt` set (Rework #5)
- [ ] **Old UI continues using the legacy edit endpoint** — direct SQL UPDATE on `program_slabs`, no MC, no Mongo write (Rework #5). Works uniformly whether the tier was originally created via old UI (legacy) or new UI (has SNAPSHOT history); the legacy flow doesn't distinguish.
- [ ] When old UI edits a tier that has a pending new-UI DRAFT, the SQL UPDATE succeeds (legacy path is unaffected). The Mongo DRAFT's `basisSqlSnapshot` now lags SQL; this is detected at approval time (see US-5).

### US-4: Tier Deletion (DRAFT Only — Soft-Delete to DELETED)
**As** a loyalty program manager,
**I want** to delete a DRAFT tier that I no longer need,
**So that** incomplete configurations are cleaned up while preserving an audit trail.

**Business Rules:**
- Only tiers in **DRAFT** status can be deleted. ACTIVE and PENDING_APPROVAL tiers CANNOT be deleted.
- Tier reordering is NOT supported. serialNumber is immutable — any attempt to change it returns 400.
- Tier retirement (stopping ACTIVE tiers) is out of scope for this pipeline run — future epic.

**Acceptance Criteria:**
- [ ] `DELETE /v3/tiers/{tierId}` soft-deletes a DRAFT tier
- [ ] Sets status to DELETED (soft-delete, preserves audit trail in MongoDB)
- [ ] Returns 409 Conflict if tier is not in DRAFT status (with message: "Only DRAFT tiers can be deleted")
- [ ] Returns 409 Conflict if tier is ACTIVE or PENDING_APPROVAL (with message: "Tier retirement is not supported in this version")
- [ ] No maker-checker flow for deletion — DRAFT tiers are pre-approval, so deletion is immediate
- [ ] Deleted tier excluded from default listing (filtered out unless ?includeDeleted=true)
- [ ] No member reassessment needed — DRAFT tiers have no members assigned

### US-5: Maker-Checker Submit for Approval
**As** a loyalty program manager,
**I want** to submit my tier changes for approval,
**So that** an approver can review before changes go live.

**Acceptance Criteria:**
- [ ] `POST /v3/tiers/{tierId}/submit` submits tier changes for approval
- [ ] Generic framework: accepts `entityType` (TIER, BENEFIT, etc.), `entityId`, payload — via the `makechecker/` package (`ApprovableEntity`, `ApprovableEntityHandler<T>`, `MakerCheckerService<T>`)
- [ ] Changes tier status from DRAFT to PENDING_APPROVAL
- [ ] Records: requestedBy (user), timestamp, change payload
- [ ] Notifies approver (via platform notification — hook interface for future implementation)
- [ ] Submission is idempotent for the same DRAFT (Rework #5): submitting an already-PENDING_APPROVAL tier returns its current state without re-processing

### US-6: Maker-Checker Approve/Reject (with Drift Detection)
**As** a loyalty platform admin,
**I want** to approve or reject tier configuration changes,
**So that** only validated changes go live AND changes are not silently applied over stale state.

**Acceptance Criteria:**
- [ ] `POST /v3/tiers/{tierId}/approve` approves tier changes
- [ ] `POST /v3/tiers/{tierId}/reject` rejects tier changes (comment required)
- [ ] On approve, the MakerCheckerService runs the SAGA:
  1. `preApprove` (incl. **drift check**) → 2. `publish` (Thrift → SQL) → 3. `postApprove` (Mongo doc transition)
- [ ] **Drift check (Rework #5)**: in `preApprove`, re-read current SQL state of the tier and compare against the DRAFT's `meta.basisSqlSnapshot`. On any drift:
  - Approval is **blocked**. Mongo DRAFT stays PENDING_APPROVAL.
  - Approver sees: *"This DRAFT is based on a state that no longer matches the live tier. SQL changed since this DRAFT was created — &lt;field&gt; is now &lt;new&gt; (was &lt;basis&gt; at draft time). Cancel or recreate this DRAFT to proceed."*
  - Drift scope (conservative): any SQL change to the tier row or its strategy fields counts as drift. Revisit if operational friction emerges.
- [ ] **Name re-check at approval (Rework #5)**: uniqueness is re-validated against SQL. Block on collision with a clear user-facing message.
- [ ] On approve (drift + name checks pass): Thrift `createSlabAndUpdateStrategies` is called; SQL `program_slabs` is updated; `updatedBy`, `approvedBy`, `approvedAt` are set on the SQL row.
- [ ] On approve post-publish: Mongo doc transitions PENDING_APPROVAL → **SNAPSHOT** (Rework #5) — audit-only record with `meta.approvedAt`, `meta.approvedBy`. SQL is the LIVE source. New UI history view reads SNAPSHOTs.
- [ ] SNAPSHOT is write-once-at-approval and is NOT reconciled with later legacy edits (Rework #5). History view labels each SNAPSHOT row: *"Approved via new UI on &lt;approvedAt&gt; by &lt;approvedBy&gt;. May differ from current state if edited via old UI after this date."*
- [ ] On reject: tier status changes PENDING_APPROVAL → DRAFT, reviewer comment required, `basisSqlSnapshot` is preserved
- [ ] Records: reviewedBy (user), timestamp, comment, approval status
- [ ] `GET /v3/approvals` lists all pending approvals for all entities
- [ ] If the Thrift publish step fails mid-SAGA (e.g., SQL `UNIQUE` constraint collision), the SAGA `onPublishFailure` hook rolls back the Mongo status and rethrows to surface a well-formed error to the approver (Rework #5 — layered defense against race conditions)

### US-7: Maker-Checker Toggle
**As** a platform admin,
**I want** to enable/disable maker-checker per program and per entity type,
**So that** different programs can have different approval requirements.

**Acceptance Criteria:**
- [ ] `isMakerCheckerEnabled(orgId, programId, entityType)` lookup determines the new-UI flow (Rework #5: MC toggle affects **new UI writes only**; old UI writes always bypass MC)
- [ ] Configuration stored in org-level settings
- [ ] When disabled: new-UI Create → SQL ACTIVE immediately (via same Thrift path, no MC wait), Edit → SQL UPDATE immediately (no DRAFT)
- [ ] When enabled: new-UI Create → Mongo DRAFT, Edit → Mongo DRAFT (on LIVE tiers, a new DRAFT doc is created; on existing DRAFTs, the doc is updated in place)
- [ ] Toggling MC on/off does not affect existing tiers in their current state
- [ ] Old UI's legacy writes are never gated by the MC toggle (Rework #5)

## Assumptions

1. No SQL status column. SQL only contains ACTIVE tiers. customer_enrollment FK unaffected. (Rework #3)
2. **SQL `program_slabs` is the single source of truth for LIVE tier state — for both legacy and new-UI-origin tiers (Rework #5).** Mongo `UnifiedTierConfig` holds only in-flight (DRAFT/PENDING_APPROVAL) and audit (SNAPSHOT) documents.
3. Legacy tier create/edit endpoints (old UI) continue to write directly to SQL and are never gated by MC (Rework #5). This pipeline does not change their behaviour.
4. The MongoDB instance used by intouch-api-v3 (EmfMongoDataSourceManager) has sufficient capacity for tier configuration documents.
5. The Thrift interface between intouch-api-v3 and emf-parent can be extended to support the new tier sync operations without breaking backward compatibility.
6. The `metadata` JSON field in ProgramSlab can continue to store per-slab metadata (colorCode, etc.) alongside the new MongoDB-based config.
7. The UI will call these APIs -- this pipeline builds the backend only.
8. Scale is bounded: ~50 tiers per program max, max 1 active DRAFT per tier (Rework #5). Envelope response overhead is small and predictable.
9. Concurrent "old UI edits mid-MC" is rare in practice; a conservative drift-block policy is acceptable (Rework #5). Revisit if operational friction emerges.
10. SNAPSHOT retention and purging policies are out of scope for this rework; SNAPSHOT docs accumulate and are surfaced by the history view with an explicit "may be stale vs. current SQL" caveat (Rework #5).

- [ ] How many active DRAFTs per tier are allowed? _(Rework #5 Q-9)_ — **Resolved via Rework #5**: At most one DRAFT or PENDING_APPROVAL per tier. Enforced by app check + Mongo partial unique index on `(orgId, programId, slabId)` where `status IN [DRAFT, PENDING_APPROVAL]`.
- [ ] How are concurrent name collisions (old UI vs new UI creating same name) prevented? _(Rework #5 Q-1b-i)_ — **Resolved via Rework #5**: Three layers — app-level check at DRAFT creation, re-check at approval, SQL `UNIQUE (program_id, name)` backstop.
- [ ] What does `parentId` reference? _(Rework #5 Q-6)_ — **Resolved via Rework #5**: `parentId` = parent tier's `slabId` (SQL `program_slabs.id`). Parent must be LIVE in SQL; a DRAFT tier cannot be a parent.
