## Current Behaviour

### How Tiers Work Today

Capillary's loyalty platform uses "slabs" (internal term for tiers) to define membership levels within a loyalty program. The current behaviour, documented at docs.capillarytech.com and verified against the codebase:

**Data Model:**
- `ProgramSlab` entity (table: `program_slabs`) stores: id, orgId, programId, serialNumber, name, description, metadata (JSON blob), createdOn. It has NO status field, NO color field, and NO inline eligibility configuration.
- `PartnerProgramSlab` extends this for multi-loyalty partner programs.
- Tier configuration (eligibility, upgrade thresholds, downgrade rules, renewal conditions) is stored as Gson-serialized JSON in strategy configuration objects, NOT in the ProgramSlab entity itself.
- `TierConfiguration` DTO aggregates: isActive flag, per-slab downgrade configs (TierDowngradeSlabConfig[]), alert configs, renewal confirmation, threshold values, and tracker references.
- `SlabMetaData` stores colorCode and is parsed from the metadata JSON.

**Eligibility & Upgrade:**
- Four eligibility criteria types: Current Points, Lifetime Points, Lifetime Purchases, Tracker Value. The same criteria type applies to ALL tiers in a program.
- Three upgrade types: Issue Points Then Upgrade, Upgrade Then Issue Points, Issue-Upgrade-Then-Issue-Remaining (controls when tier evaluation happens during transaction processing).
- `SlabUpgradeMode` enum: EAGER (before awarding), DYNAMIC (during awarding), LAZY (after awarding).
- Strategies: `SlabUpgradeStrategy` interface with `SlabUpgradeStrategyImpl` and `ThresholdBasedSlabUpgradeStrategyImpl` implementations.

**Downgrade & Renewal:**
- Downgrade targets: SINGLE (one tier below), THRESHOLD (to matching tier), LOWEST (base tier).
- Downgrade conditions: purchase amount, visit count, points, tracker values, with AND/OR logic.
- Renewal condition types: Any (N-1), All, Custom (AND/OR with parentheses).
- PEB (Points Engine Backend) handles bulk tier downgrade/reassessment via `TierDowngradeBatchServiceImpl`, `TierReassessmentServiceImpl`, and various `GapToUpgradeCalculator` implementations.

**APIs:**
- NO REST API endpoints exist for tier CRUD in intouch-api-v3 (verified: zero controllers/facades reference tiers/slabs).
- All current tier operations go through Thrift (EMFThriftServiceImpl) or internal service calls.
- Tier creation uses `PointsEngineRuleService.createSlabAndUpdateStrategies()` which creates the ProgramSlab record AND updates associated strategy configurations.

**Maker-Checker (Approval Framework):**
- Generic approval framework exists in makechecker package for unified promotions (UnifiedPromotionFacade, ApprovalStatus, PromotionReviewRequest).
- No approval flow exists yet for tier configuration.
- The promotion approval pattern uses: MongoDB draft storage (`@Document(collection = "unified_promotions")`), `PromotionStatus` lifecycle (DRAFT -> PENDING_APPROVAL -> ACTIVE -> PAUSED -> STOPPED), `EntityOrchestrator` with ApprovableEntityHandler pattern to sync MongoDB -> SQL on approval.

**UI:**
- Current TiersView.js in Garuda/cap-loyalty-ui renders a flat list of tier cards from a single API. No comparison, no configuration entry, no KPI data, no status visibility, no action surface.

### Constraints from Current System
- Tiers must be created in sequential order (cannot insert between existing tiers).
- Existing tiers cannot be deleted (per docs).
- The `serialNumber` ordering is used by upgrade/downgrade logic and must remain stable.
- `customer_enrollment` table tracks each member's `current_slab_id` -- any change to tier structure must preserve referential integrity.

## Problem Statement

Loyalty program managers cannot efficiently manage tier configurations because the experience is fragmented across six screens, tier logic is invisible, and there are no APIs for programmatic tier management. Creating a single tier takes ~45 minutes. There is no comparison view, no lifecycle management, no approval workflow, and no way to see impact before committing changes.

---

## Goals

1. **Tier CRUD APIs**: Self-reliant REST APIs for listing, creating, editing, and deleting tiers -- with validation in the API, not the UI.
2. **Comparison Matrix Data**: A single API call returns all tiers in a program with full configuration across all dimensions (eligibility, renewal, downgrade, benefits linkage).
3. **Generic Maker-Checker Framework**: A reusable approval workflow that tiers plug into first, with benefits, subscriptions, and other entities plugging in later.
4. **Tier Lifecycle Management**: Status-based lifecycle (Draft, Pending Approval, Active, Deleted) with versioned edits and rollback capability. No PAUSED or STOPPED states. Tier retirement (stopping ACTIVE tiers) is deferred to a future epic.
5. **Extensible Architecture**: Design supports future change log (E1-US5), simulation mode (E1-US6), tier retirement, and benefits module (E2) without rework.
6. **Tier Ordering Immutability**: Tier reordering is explicitly not supported. serialNumber is auto-assigned and immutable.
7. **Unified Read Surface (Rework #5)**: The new `/v3/tiers` API returns **all** tiers of a program — both legacy SQL-origin tiers and new-UI-origin tiers. Legacy tiers surface through a read-only SQL→DTO converter; no bootstrap migration into Mongo.
8. **Dual Write Paths, Single MC Scope (Rework #5)**: Maker-checker applies **if-and-only-if** the write originated from the new UI. Old UI writes continue via the legacy direct-SQL path with no MC. New UI writes always go through Mongo DRAFT → MC → Thrift → SQL.
9. **Drift Detection Before Approval (Rework #5)**: Each Mongo DRAFT captures a basis SQL snapshot at draft-creation time. At approval time, the current SQL state is compared against the basis; any drift blocks the approval with a clear user-facing message.
10. **SQL as Single Source of Truth for LIVE (Rework #5)**: The live state of every tier — legacy or new-UI-origin — is always read from SQL `program_slabs`. Mongo holds in-flight DRAFT/PENDING_APPROVAL and audit-only SNAPSHOT docs. SNAPSHOTs are write-once-at-approval and never reconciled with later legacy edits.

**As** a loyalty program manager,
**I want** to retrieve all tiers in my program with full configuration displayed in a comparable format,
**So that** I can understand the tier structure, identify gaps, and spot inconsistencies from a single API call.

### US-6: Maker-Checker Approve/Reject (with Drift Detection)
**As** a loyalty platform admin,
**I want** to approve or reject tier configuration changes,
**So that** only validated changes go live AND changes are not silently applied over stale state.

**Acceptance Criteria:**
- [ ] `POST /v3/tiers/{tierId}/approve` approves tier changes
- [ ] `POST /v3/tiers/{tierId}/reject` rejects tier changes (comment required)
- [ ] On approve, the MakerCheckerService runs the SAGA:
  1. `preApprove` (incl. **drift check**) → 2. `publish` (Thrift → SQL) → 3. `postApprove` (Mongo doc transition)
- [ ] **Drift check (Rework #5)**: in `preApprove`, re-read current SQL state of the tier and compare against the DRAFT's `meta.basisSqlSnapshot`. On any drift:
  - Approval is **blocked**. Mongo DRAFT stays PENDING_APPROVAL.
  - Approver sees: *"This DRAFT is based on a state that no longer matches the live tier. SQL changed since this DRAFT was created — &lt;field&gt; is now &lt;new&gt; (was &lt;basis&gt; at draft time). Cancel or recreate this DRAFT to proceed."*
  - Drift scope (conservative): any SQL change to the tier row or its strategy fields counts as drift. Revisit if operational friction emerges.
- [ ] **Name re-check at approval (Rework #5)**: uniqueness is re-validated against SQL. Block on collision with a clear user-facing message.
- [ ] On approve (drift + name checks pass): Thrift `createSlabAndUpdateStrategies` is called; SQL `program_slabs` is updated; `updatedBy`, `approvedBy`, `approvedAt` are set on the SQL row.
- [ ] On approve post-publish: Mongo doc transitions PENDING_APPROVAL → **SNAPSHOT** (Rework #5) — audit-only record with `meta.approvedAt`, `meta.approvedBy`. SQL is the LIVE source. New UI history view reads SNAPSHOTs.
- [ ] SNAPSHOT is write-once-at-approval and is NOT reconciled with later legacy edits (Rework #5). History view labels each SNAPSHOT row: *"Approved via new UI on &lt;approvedAt&gt; by &lt;approvedBy&gt;. May differ from current state if edited via old UI after this date."*
- [ ] On reject: tier status changes PENDING_APPROVAL → DRAFT, reviewer comment required, `basisSqlSnapshot` is preserved
- [ ] Records: reviewedBy (user), timestamp, comment, approval status
- [ ] `GET /v3/approvals` lists all pending approvals for all entities
- [ ] If the Thrift publish step fails mid-SAGA (e.g., SQL `UNIQUE` constraint collision), the SAGA `onPublishFailure` hook rolls back the Mongo status and rethrows to surface a well-formed error to the approver (Rework #5 — layered defense against race conditions)

## Assumptions

1. No SQL status column. SQL only contains ACTIVE tiers. customer_enrollment FK unaffected. (Rework #3)
2. **SQL `program_slabs` is the single source of truth for LIVE tier state — for both legacy and new-UI-origin tiers (Rework #5).** Mongo `UnifiedTierConfig` holds only in-flight (DRAFT/PENDING_APPROVAL) and audit (SNAPSHOT) documents.
3. Legacy tier create/edit endpoints (old UI) continue to write directly to SQL and are never gated by MC (Rework #5). This pipeline does not change their behaviour.
4. The MongoDB instance used by intouch-api-v3 (EmfMongoDataSourceManager) has sufficient capacity for tier configuration documents.
5. The Thrift interface between intouch-api-v3 and emf-parent can be extended to support the new tier sync operations without breaking backward compatibility.
6. The `metadata` JSON field in ProgramSlab can continue to store per-slab metadata (colorCode, etc.) alongside the new MongoDB-based config.
7. The UI will call these APIs -- this pipeline builds the backend only.
8. Scale is bounded: ~50 tiers per program max, max 1 active DRAFT per tier (Rework #5). Envelope response overhead is small and predictable.
9. Concurrent "old UI edits mid-MC" is rare in practice; a conservative drift-block policy is acceptable (Rework #5). Revisit if operational friction emerges.
10. SNAPSHOT retention and purging policies are out of scope for this rework; SNAPSHOT docs accumulate and are surfaced by the history view with an explicit "may be stale vs. current SQL" caveat (Rework #5).
