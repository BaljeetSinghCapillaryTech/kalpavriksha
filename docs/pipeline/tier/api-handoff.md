### Two controllers, two behaviours

Tier endpoints are served by two separate `@RestController` classes sharing the
same `@RequestMapping("/v3/tiers")` base path:

- **`TierController`** (maker/CRUD) — endpoints 1–5. Has **local
  `@ExceptionHandler` methods** (`TierController.java` L129–L150) that override
  the global advice. `NotFoundException → 404`, `ConflictException → 409`.
- **`TierReviewController`** (reviewer workflow) — endpoints 6–8. Has **NO
  local handlers**. Inherits the global `TargetGroupErrorAdvice` mapping.

This distinction matters because the global advice has a known quirk for
`NotFoundException` — see §7.1 and §8.1.

## 2. Transport, auth and tenancy

### 2.1 Auth

Every endpoint takes an `AbstractBaseAuthenticationToken token` argument
(`TierController.java` L56, L67, L82, L96, L108; `TierReviewController.java`
L41, L63, L84). The controller derives two values from it:

- `orgId = token.getIntouchUser().getOrgId()` — tenant scoping, passed to every
  facade call. `orgId` is a primitive `long` (see `TierFacade.listTiers`
  signature at `TierFacade.java` L131 and the javadoc at L123–L129 explaining
  that `orgId=0` is not a wildcard).
- `userId = String.valueOf(token.getIntouchUser().getEntityId())` — stamped on
  `TierMeta.createdBy` / `updatedBy` (`TierController.java` L84, L98, L110).

The reviewer endpoint also reads `user.getTillName()` for `reviewedBy`
stamping on approve/reject (`TierReviewController.java` L69, L72).

The UI does not pass `orgId` on the query string or body — it is always
resolved from the authenticated session. Passing a different `orgId` from the
client has no effect.

### 2.2 Content negotiation

- Request: `Content-Type: application/json` required on bodies (POST/PUT).
- Response: `application/json` always. Create is explicitly typed
  (`@PostMapping(produces = "application/json")`, `TierController.java` L78).
- Bodies that fail to parse as JSON return **HTTP 400** with body code
  `"COMMON.INVALID_INPUT"` — see `TargetGroupErrorAdvice.java`
  `handleHttpMessageNotReadable` L169–L180.

### 2.3 Idempotency-Key (accepted, not honoured)

`POST /v3/tiers` accepts an optional `Idempotency-Key` request header
(`TierController.java` L81). **The value is read into a local variable but
never used in `createTier`** — no dedup store, no replay detection. Sending
the same key twice creates two tiers. Treat the header as reserved for a
future epic; do not rely on it for safe retry today.

## 3. Response envelope

Every 2xx and most 4xx responses wrap the payload in
`com.capillary.intouchapiv3.models.ResponseWrapper` (`ResponseWrapper.java`):

```json
{
  "data":     <T | null>,
  "errors":   [{ "code": <long | null>, "message": <string> }] | null,
  "warnings": [{ "message": <string> }] | null
}
```

- `data` is the typed payload on success; on most error paths it is `null`.
  **One important exception** — drift errors (§5.7.1) put a structured
  `{code, diffs}` object in `data` even on 409.
- `errors[*].code` is typed `Long` — it is `null` on most flows and carries a
  numeric message code only when `CommonUtils.getErrorMessageCode(...)` maps
  the exception's message key (e.g. validation failures). Do not hardcode
  specific numbers; pattern-match on `errors[0].message` where appropriate.
- `warnings` is populated by downstream flows but **no tier endpoint sets it
  today** (always `null`).

### 4.2 `TierEnvelope` — the read shape

The read endpoints (`GET /v3/tiers` and `GET /v3/tiers/{tierId}`) do **not**
return `UnifiedTierConfig` directly. They return a `TierEnvelope` that pairs
the SQL LIVE state with the Mongo workflow-visible state:

```json
{
  "slabId":         3850,
  "live":           { /* TierView — built from SQL */ } | null,
  "pendingDraft":   { /* TierView — built from Mongo DRAFT or PENDING_APPROVAL */ } | null,
  "origin":         "LEGACY_SQL_ONLY" | "MONGO_ONLY" | "BOTH",
  "hasPendingDraft": true | false
}
```

Evidence: `TierEnvelope.java` — class-level `@JsonInclude(JsonInclude.Include.NON_NULL)`,
fields `slabId`, `live`, `pendingDraft`, `origin`, plus derived
`hasPendingDraft()` tagged `@JsonProperty("hasPendingDraft")`.

**Six scenarios the UI must handle** (from
`TierEnvelopeBuilder.java` `build()` / `buildOne()`):

| # | `origin` | `live` | `pendingDraft` | `hasPendingDraft` | What it is |
|---|---|---|---|---|---|
| 1 | `BOTH` | populated | `null` | `false` | LIVE with no pending edit |
| 2 | `MONGO_ONLY` | `null` | populated | `true` | Brand-new tier in DRAFT/PENDING (no SQL row yet) |
| 3 | `BOTH` | populated | populated (`draftStatus=DRAFT`) | `true` | Edit-of-LIVE, not yet submitted |
| 4 | `BOTH` | populated | populated (`draftStatus=PENDING_APPROVAL`) | `true` | Edit-of-LIVE, awaiting approval |
| 5 | `LEGACY_SQL_ONLY` | populated | `null` | `false` | Legacy tier with no Mongo doc |
| 6 | — | — | — | — | SNAPSHOT / DELETED / PUBLISH_FAILED Mongo docs are filtered out — they never surface as envelopes |

Because of `@JsonInclude(NON_NULL)`, **null fields are absent from the wire**.
UI code should test `envelope.hasPendingDraft` and `envelope.live != null`,
not `envelope.pendingDraft === null`.

### 4.5 `TierStatus` (enum)

```
DRAFT, PENDING_APPROVAL, ACTIVE, DELETED, SNAPSHOT, PUBLISH_FAILED
```

Evidence: `TierStatus.java` — all six values.

Meaning:

- `DRAFT` — maker editing; mutable via PUT; can be submitted or deleted.
- `PENDING_APPROVAL` — awaiting reviewer; mutable via PUT (edit-in-place) but
  should be treated as locked by the UI maker seat.
- `ACTIVE` — published to SQL `program_slabs`; PUT on it creates a *versioned
  draft* with `parentId` pointing back to the ACTIVE — the ACTIVE row is not
  touched.
- `DELETED` — soft-deleted (DRAFT only); history is retained.
- `SNAPSHOT` — historical copy left behind when a versioned draft is
  approved, so the prior ACTIVE row is preserved for audit. Never surfaces in
  envelopes.
- `PUBLISH_FAILED` — SAGA failure state. See §8.4.

### 5.2 `GET /v3/tiers/{tierId}` — detail

`tierId` is the Mongo `objectId` (preferred), or `tierUniqueId`
(`TierFacade.java` L327–L368).

**Success — 200 OK, `ResponseWrapper<TierEnvelope>`**

```json
{
  "data": { /* TierEnvelope */ },
  "errors": null,
  "warnings": null
}
```

Lookup rules (`TierFacade.getTierDetail`):

1. Try Mongo by `objectId`. If hit and status is DRAFT or PENDING_APPROVAL, surface as `pendingDraft`. If hit but status is ACTIVE/SNAPSHOT/DELETED/PUBLISH_FAILED, try the paired SQL LIVE row by `slabId`; if present, return a LIVE-only envelope; otherwise 404.
2. Try Mongo by `tierUniqueId`.
3. If `tierId` parses as a numeric `slabId`, **still 404** — see §8.5.

**Errors**

- **`404`** — Not found. **`ConflictException → 409` is impossible on this
  endpoint (no conflict path).** Emitted by the local
  `@ExceptionHandler(NotFoundException.class)` in `TierController.java` L129–L135:

  ```json
  {
    "data": null,
    "errors": [ { "code": 404, "message": "Tier not found: <tierId>" } ],
    "warnings": null
  }
  ```
- `401/403` — auth.
- `500` — unexpected.

|---|---|

**Errors** (via global advice — note §8.1):

| Status | When | Notes |
|---|---|---|

### 5.7 `POST /v3/tiers/{tierId}/approve` — reviewer decision

Served by `TierReviewController`.

**Request body** — `Map<String, String>`:

```json
{ "approvalStatus": "APPROVE" | "REJECT", "comment": "optional review note" }
```

`comment` is optional; `approvalStatus` is case-insensitive (facade uses
`equalsIgnoreCase`, `TierFacade.java` L422–L430).

**Success — 200 OK, `ResponseWrapper<UnifiedTierConfig>`**

On `APPROVE`: runs the SAGA — pre-approve (drift check + name re-uniqueness),
publish via Thrift `createSlabAndUpdateStrategies` (with SLAB_UPGRADE and
SLAB_DOWNGRADE strategies), post-approve (status → ACTIVE, stamp `slabId`,
parent → SNAPSHOT).

On `REJECT`: status → DRAFT, stamp `rejectedBy` / `rejectedAt` /
`rejectionComment`.

**Errors**

| Status | When | Body / notes |
|---|---|---|
| **`200`** | Tier not found. | Same NotFoundException quirk as §5.6. See §8.1. |
| `409` | Tier not in PENDING_APPROVAL. | `"Only PENDING_APPROVAL tiers can be approved/rejected. Tier is in <status> status."` (`TierFacade.java` L418–L420). Handled by global advice `handleConflictException`. |
| `409` | **APPROVAL_BLOCKED_DRIFT** — SQL LIVE row has drifted since the draft's basis snapshot. Structured diff response — see §5.7.1. | |
| `500` | `approvalStatus` is neither `APPROVE` nor `REJECT` — raises `IllegalArgumentException` which falls through to global `handleThrowable` → 500 with the generic `"Something went wrong, please try after sometime."` message. **Validate on the client before sending.** | `TierFacade.java` L429 + `TargetGroupErrorAdvice.java` L306–L317. |
| `500` | SAGA publish failure (Thrift RPC error). Tier is transitioned to `PUBLISH_FAILED` and best-effort saved before the exception propagates. | `MakerCheckerService.approve` L64–L76. |

#### 5.7.1 Drift error body (APPROVE only)

When `TierApprovalHandler.preApprove` detects that the LIVE SQL row has been
modified since the draft's `basisSqlSnapshot` was captured, it throws
`TierApprovalDriftException`. The dedicated `TierErrorAdvice` maps this to
**`409 Conflict`** with a structured body — note the diffs go into `data`, not
`errors`:

```json
{
  "data": {
```

**Query params** — `programId` (required, int).

with `status = PENDING_APPROVAL` for the org+program (`TierFacade.java`
L436–L439). Returns `UnifiedTierConfig` documents (not envelopes) — this
endpoint is the reviewer's raw queue.

**Errors** — 400/401/403/500 via global advice.

## 7. Error handling — the full picture

### 7.1 Handler precedence

Spring resolves `@ExceptionHandler` methods declared **on the controller
class** before methods declared on `@ControllerAdvice`. Because `TierController`
declares local handlers for `NotFoundException` and `ConflictException`
(`TierController.java` L129–L150), tier CRUD endpoints get the correct
status codes. `TierReviewController` does **not** declare any — the review
endpoints fall through to the global `TargetGroupErrorAdvice`.

### 7.2 Global advice — `TargetGroupErrorAdvice` mapping

Authoritative table, drawn from `TargetGroupErrorAdvice.java`:

| Exception | HTTP status | Body `errors[0].code` | Notes |
|---|---|---|---|
| `NotFoundException` | **`200 OK`** | mapped via `CommonUtils.getErrorMessageCode` | L74–L77. **BUG** — see §8.1. |
| `InvalidInputException` | `400` | mapped | L47–L72. Validators throw this. |
| `ConflictException` | `409` | mapped | L79–L94. |
| `HttpMessageNotReadableException` | `400` | `COMMON.INVALID_INPUT` | L169–L180. Bad JSON body. |
| `MethodArgumentNotValidException` | `400` | per-field resolver codes | L144–L167. `@Valid` on DTO. |
| `OperationFailedException` | `500` | mapped | |
| `EMFThriftException` | `500` | mapped | Downstream Thrift failure. |
| `ServiceException` | `400` | mapped | |
| `ConstraintViolationException` | `400` | mapped | JSR-303 at service layer. |
| `DataIntegrityViolationException` | `400` | `500` (note the mismatch: status=400, code=500) | Legacy quirk — keep if you see it. |
| `AccessDeniedException` | `403` | mapped | |
| `BadCredentialsException` | `401` | mapped | |
| `TokenExpiredException` | `498` | mapped | |
| `Throwable` (fall-through) | `500` | `"Something went wrong, please try after sometime."` | L306–L317. |

### 7.3 Validation messages from `TierCreateRequestValidator`

These are the exact message strings thrown as `InvalidInputException` and
therefore surfaced under `errors[0].message` with HTTP 400. **Codes
`9001`–`9009` are defined as `public static final int` constants in
`TierCreateRequestValidator.java` but are NEVER used in any `throw` call** —
they are dead constants. Do not rely on the numeric `code` field matching a
specific validator check.

| Check | Exact thrown message |
|---|---|
| Body null | `"Request body is required"` |
| `programId` null | `"programId is required"` |
| `name` blank | `"name is required"` |
| `name` length > 100 | `"name must not exceed 100 characters"` |
| `description` length > 500 | `"description must not exceed 500 characters"` |
| `color` not `#RRGGBB` | `"color must be hex format #RRGGBB"` |
| `kpiType` not in allowed set | `"kpiType must be one of: [CURRENT_POINTS, LIFETIME_POINTS, LIFETIME_PURCHASES, TRACKER_VALUE, PURCHASE, VISITS, POINTS, TRACKER]"` |
| `threshold` ≤ 0 | `"threshold must be positive"` |
| `upgradeType` not in allowed set | `"upgradeType must be one of: [EAGER, DYNAMIC, LAZY, IMMEDIATE, SCHEDULED]"` |

### 7.4 Error body templates

**Standard error (most 4xx/5xx, via global advice)**
```json
{
  "data": null,
  "errors": [ { "code": null | <long>, "message": "<english message>" } ],
  "warnings": null
}
```

**Local-handler 404 (TierController)** — same shape, `code = 404`.
**Local-handler 409 (TierController)** — same shape, `code = 409`.

**Field validation (MethodArgumentNotValidException)** — `errors` array may
contain multiple entries, one per failing field, with message codes resolved
via `resolverService`.

**Drift 409** — non-standard: `data` is populated with `{code, diffs}` in
addition to the `errors` array. See §5.7.1.

## 8. Known issues and gotchas (READ THIS)

These are real behaviours in the current code. Some are bugs, some are
intentional compromises. Until they are fixed, the UI must handle them.

### 8.1 `NotFoundException → HTTP 200` on review endpoints

`TargetGroupErrorAdvice.handleNotFoundException` (L74–L77) returns
`error(OK, e)` — HTTP 200 with an error object in `errors[0]` and `data = null`.

`TierController` **overrides** this locally → proper 404s for
`GET/PUT/DELETE /v3/tiers/{tierId}`.

`TierReviewController` does **not** override → submit/approve return HTTP 200
with an `errors` payload when the tier id is wrong. **UI must inspect
`errors` on 200 for `/submit` and `/approve`.** Treat:

```json
{ "data": null, "errors": [ { "code": ..., "message": "Tier not found: ..." } ], "warnings": null }
```

…as a not-found condition even on 200.

### 8.2 `GET /v3/tiers?status=…` returns an empty list

`TierFacade.listTiers` L132–L144: if `statusFilter != null && !statusFilter.isEmpty()`,
the method short-circuits to an empty envelope list with all-zero KPI summary.
This path exists to keep an "admin / legacy" hook available without
compromising the envelope contract. The UI should **not** send `status` on
this endpoint — omit it to get the normal envelope listing.

### 8.3 `totalMembers` can be `null`

`KpiSummary.totalMembers` is typed `Integer` and is explicitly returned as
`null` when any envelope in the result is `LEGACY_SQL_ONLY` — because member
counts live on the Mongo side only and silently returning 0 would be a lie
(`TierFacade.computeTotalMembers` L213–L224). Render `null` as `—` / `n/a`,
not `0`.

### 8.4 `PUBLISH_FAILED` is a real status you'll see

On APPROVE the SAGA can fail at the Thrift publish step. The maker-checker
service does a best-effort save of the tier with `status = PUBLISH_FAILED`
before the exception propagates (`MakerCheckerService.approve` L64–L76). The
HTTP response is 500 with the generic message, **and** next time the UI loads
the list, that tier may appear in Mongo with `PUBLISH_FAILED`. Because
envelopes only surface DRAFT/PENDING_APPROVAL Mongo docs (§4.2 scenario 6),
`PUBLISH_FAILED` tiers **do not appear in list responses** — they will resurface
if retrieved directly by detail with a matching SQL row, otherwise are
invisible to the user. This is a known gap; a recovery flow is deferred.

### 8.5 Detail by numeric `slabId` returns 404

`TierFacade.getTierDetail` L356–L365: when the `{tierId}` path segment parses
as a `Long`, the facade would need to look up the SQL row by `slabId` — but
`SqlTierReader.readLiveTierForSlab` requires `programId`, which the detail
endpoint does not receive. Rather than guess, the method returns 404. To
detail-view a legacy SQL-only tier today, fetch it through the list endpoint.

### 8.6 Unknown `approvalStatus` is a 500, not a 400

`TierFacade.handleApproval` L422–L430 accepts `"APPROVE"` / `"REJECT"`
(case-insensitive). Any other value raises `IllegalArgumentException` —
unmapped by the global advice, so it falls through to `handleThrowable`
(L306–L317) and returns HTTP 500 with the generic server-error message.
**Validate on the client before calling** — the server will not help you
here.

### 8.7 Concurrency / race windows on the list endpoint

The three round-trips in `listTiers` are not transactional (`TierFacade.java`
L95–L122 documents four observable races). Net effect for the UI:

- A tier just published may be missing for one poll cycle.
- A newly-created DRAFT always appears (good).
- A just-deleted DRAFT may appear in the list; a subsequent edit/approve call
  on it will 404 (submit/approve returns 200 with errors — §8.1). UI must
  defensively handle "id from last list no longer exists".

### 8.8 `rejectionComment` appears on a DRAFT envelope's `pendingDraft` view

When a reviewer rejects, the tier transitions back to `DRAFT` and
`meta.rejectionComment` is stamped. The next list poll surfaces the doc as
`pendingDraft` with `draftStatus = DRAFT` and `rejectionComment` populated.
This is intentional — the maker seat renders the rejection reason next to
the editable draft.

### 8.9 `engineConfig` is on `UnifiedTierConfig` but NOT on `TierView`

`TierView` deliberately omits `engineConfig` (`TierView.java` — no such
field). The list and detail endpoints never expose engine internals. If you
see `engineConfig` on a payload, you are looking at a create/update/approve
response (`UnifiedTierConfig`), not a read response (`TierEnvelope`).

### 8.10 `basisSqlSnapshot` is server-internal

Exposed on `TierMeta` for debuggability but carries no UI meaning. It is a
frozen SQL row used by `TierDriftChecker` at approval time. Do not render it,
do not diff against it from the client.

### 8.11 `Idempotency-Key` header is reserved, not implemented

See §2.3. The server accepts the header and drops it. Rely on the client's
own at-least-once retry behaviour plus the 409-on-duplicate-name check to
prevent accidental duplicates.

### 8.12 Validator codes 9001–9009 are dead

Declared in `TierCreateRequestValidator` but never reach the wire. `errors[0].code`
will usually be `null` or a `MessageSource`-resolved number, never 9001–9009.

## 10. UI handoff checklist

1. **Treat `TierEnvelope`, not `UnifiedTierConfig`, as the read shape.** Tests
   that assumed flat documents from `GET /v3/tiers` (v2) need to be updated.
2. **Handle six envelope scenarios** (§4.2). Never assume both `live` and
   `pendingDraft` are present; never assume only one is present either.
3. **Check `errors[0]` on HTTP 200 responses from `/submit` and `/approve`.**
   Not-found returns 200 with an error object (§8.1).
4. **Omit the `status` query param on `GET /v3/tiers`** or you'll get an
   empty list (§8.2). Filter client-side if you need status subsets.
5. **Render `totalMembers === null` as `—` or `n/a`.** It is not zero (§8.3).
6. **Validate `approvalStatus` is `APPROVE` or `REJECT` on the client.**
   Anything else is a 500 (§8.6).
7. **On drift 409, render `data.diffs`, not `errors[0].message`.** The
   structured diff is the source of truth (§5.7.1).
8. **Store `objectId` as the primary tier handle** — `tierUniqueId` works too,
   but numeric `slabId` does not route correctly through the detail endpoint
   (§8.5).
9. **Do not send the `Idempotency-Key` header and expect dedup.** Use it as
   a client-side correlation id only (§2.3).
10. **Do not render `engineConfig` or `basisSqlSnapshot`.** These are engine
    internals leaked through the document, not part of the visible contract
    (§8.9, §8.10).
