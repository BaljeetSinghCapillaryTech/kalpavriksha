Tiers CRUD — API Reference

Complete API contract for the Garuda UI team. Covers tier CRUD, generic maker-checker approval workflows, and all supporting endpoints.

Base URL https://{host}/v3
Auth Bearer token
Format JSON
Endpoints 11

List tiers with KPI summary

Authentication

All endpoints require a valid Bearer token in the Authorization header.

Authorization: Bearer <access_token>
ℹ
The token carries orgId, entityId (user ID), and tillName. The orgId is extracted server-side — you never pass it explicitly.

Response Envelope

Every response wraps data in ResponseWrapper<T>.

SUCCESS
{
  "data": { ... },
  "errors": null,
  "warnings": null
}
ERROR
{
  "data": null,
  "errors": [
    { "code": 400,
      "message": "Name is required" }
  ],
  "warnings": null
}

Status Lifecycle

Tier statuses and allowed transitions. Use for badge rendering and button visibility.

DRAFT PENDING_APPROVAL ACTIVE STOPPED SNAPSHOT

State Machine

stateDiagram-v2 [*] --> DRAFT : Create (MC on) [*] --> ACTIVE : Create (MC off) DRAFT --> PENDING_APPROVAL : Submit DRAFT --> [*] : Delete (immediate) PENDING_APPROVAL --> ACTIVE : Approve PENDING_APPROVAL --> DRAFT : Reject ACTIVE --> STOPPED : Stop (MC off) ACTIVE --> DRAFT : Edit (new version) ACTIVE --> SNAPSHOT : Version replaced STOPPED --> [*] : Terminal SNAPSHOT --> [*] : Terminal

Status Reference

StatusBadgeUser Can...
DRAFTDRAFTEdit, Submit for Approval, Delete
PENDING_APPROVALPENDINGView only (approver: Approve / Reject)
ACTIVEACTIVEEdit (creates new version), Stop
STOPPEDSTOPPEDView only (terminal)
SNAPSHOTSNAPSHOTView only (archived version)

POST /v3/maker-checker/{changeId}/approve

Approves a pending change. Triggers TierChangeApplier to sync MongoDB to SQL via Thrift.

REQUEST BODY
{
  "comment": "Approved. Platinum tier config looks good."
}
✓
Side effects on approve:
• Tier: PENDING_APPROVAL → ACTIVE
• SQL: ProgramSlab created + strategies updated
• metadata.sqlSlabId populated
• If versioned edit: old ACTIVE → SNAPSHOT
POST /v3/maker-checker/{changeId}/reject

Rejects a pending change. Comment is required. Tier reverts to DRAFT status.

REQUEST BODY
{
  "comment": "Threshold too low -- 1200 RM overlaps with Silver. Increase to 1500+."
}
✗
Comment is required. The API returns 400 if comment is missing on reject.

Flow: Create + Submit + Approve

Complete lifecycle of creating a new tier with MC enabled.

sequenceDiagram participant UI as Garuda UI participant API as /v3/tiers participant MC as /v3/maker-checker participant SQL as MySQL (via Thrift) UI->>API: POST /v3/tiers {Platinum config} API-->>UI: 201 {status: DRAFT, objectId: 661b...} UI->>MC: POST /submit {entityType: TIER, entityId: 661b...} MC-->>UI: 200 {changeId: 661e..., status: PENDING_APPROVAL} Note over UI: Tier status: PENDING_APPROVAL UI->>MC: POST /661e.../approve {comment: "Looks good"} MC->>SQL: Thrift sync (ProgramSlab + strategies) MC-->>UI: 200 {status: APPROVED} Note over UI: Tier status: ACTIVE, sqlSlabId set
Create Tier

POST /v3/tiers → returns DRAFT with objectId

Submit for Approval

POST /v3/maker-checker/submit → tier becomes PENDING_APPROVAL, returns changeId

Approve

POST /v3/maker-checker/{changeId}/approve → syncs to SQL, tier becomes ACTIVE

Verify

GET /v3/tiers?programId=977 → Platinum appears with status: ACTIVE and sqlSlabId set

Flow: Edit ACTIVE Tier (Versioned)

Editing an ACTIVE tier creates a new DRAFT. The ACTIVE version stays live until approved.

sequenceDiagram participant UI as Garuda UI participant API as /v3/tiers UI->>API: GET /v3/tiers/661a...9e (fetch for edit form) API-->>UI: 200 {Gold, ACTIVE, engineConfig: {...}} UI->>API: PUT /v3/tiers/661a...9e {name: "Gold Plus"} API-->>UI: 200 {NEW objectId: 661c..., DRAFT, parentId: 661a...9e} Note over UI: Gold (ACTIVE) + Gold Plus (DRAFT) both visible Note over UI: Submit + Approve... Note over UI: Gold Plus → ACTIVE, Gold → SNAPSHOT
ℹ
Zero downtime. The ACTIVE version serves live traffic until the DRAFT is approved. On approval, the swap is atomic: new → ACTIVE, old → SNAPSHOT.

Flow: Delete (Draft vs Active)

Delete behavior depends on the tier's current status.

Deleting a DRAFT

DELETE /v3/tiers/661b5a...
→ 204 No Content (immediate)

No MC gate. The tier was never live. Document removed from MongoDB.

Deleting an ACTIVE (MC on)

DELETE /v3/tiers/661a3f...
→ 200 OK (PendingChange created)

Tier stays ACTIVE until approver approves the stop. Then: STOPPED + SQL sync.

Not In Scope (This Release)

These features are not available. Do not build UI for them.

Tier Reorder

serialNumber is immutable and auto-assigned. Tiers cannot be reordered or inserted between existing tiers. No current plan.

Tier Settings (Program-Level)

The "Tier Settings" button maps to program-level slab settings (upgrade mode, point category). API not designed yet. Future pipeline run.

Version History / Diff

SNAPSHOT documents are preserved in MongoDB. A dedicated history endpoint is not built. Planned for Change Log (E1-US5).

Bulk Operations

No batch create/update/delete. One tier at a time.

Real-time Member Counts

memberStats.memberCount is cached (refreshed every ~10 min via cron). No live query or manual refresh endpoint.
