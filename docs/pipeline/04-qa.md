## 2. Test Scenarios

Each scenario is numbered TS-01 through TS-72.
Format: **Name**, Category, Description, Pre-conditions, Steps, Expected Outcome.

### 2.1 Happy Path

---

**TS-01: List tiers — program with 3 ACTIVE tiers**
- **Category**: Happy Path
- **Pre-conditions**: Org 100, Program 200 exists with 3 ACTIVE tiers (serialNumber 1, 2, 3) in MongoDB + MySQL
- **Steps**: `GET /v3/tiers?programId=200` with valid token (orgId=100)
- **Expected**:
  - HTTP 200
  - `data` contains exactly 3 `TierResponse` objects
  - Ordered by serialNumber ascending (base first)
  - Each response includes: name, description, colorCode, serialNumber, status=ACTIVE, eligibility, validity, renewal, downgrade, upgradeBonus, memberCount (Long), createdOn (ISO-8601), lastModifiedOn (ISO-8601)
  - `memberCount` is not null for ACTIVE tiers
  - `slabId` is populated (non-null) for ACTIVE tiers
  - Response envelope: `{"data": [...], "errors": [], "warnings": []}`

---

**TS-02: List tiers — program with mix of statuses (DRAFT, PENDING, ACTIVE, STOPPED)**
- **Category**: Happy Path
- **Pre-conditions**: Program 200 has 4 tiers: 1 ACTIVE, 1 DRAFT, 1 PENDING_APPROVAL, 1 STOPPED
- **Steps**: `GET /v3/tiers?programId=200` (default: includeInactive=false)
- **Expected**:
  - HTTP 200
  - Only ACTIVE tier returned in `data` (DRAFT, PENDING_APPROVAL, STOPPED excluded)
  - DRAFT/PENDING tiers have `slabId=null` and `memberCount=null` — verified by absence in default response
  - **Note**: STOPPED tiers excluded confirms soft-delete exclusion logic

---

**TS-03: List tiers — ordering by serialNumber**
- **Category**: Happy Path
- **Pre-conditions**: Program 200 has tiers with serialNumbers 1, 3, 5 (gaps from soft-delete)
- **Steps**: `GET /v3/tiers?programId=200`
- **Expected**:
  - HTTP 200
  - Tiers returned in order: serialNumber 1, 3, 5 (ascending)
  - Serial number gaps are accepted — no renumbering (ADR-07 / R-7 resolution)

---

**TS-04: List tiers — soft-deleted tier excluded by default**
- **Category**: Happy Path / AC-1-4
- **Pre-conditions**: Program 200 has 2 ACTIVE tiers, 1 STOPPED tier
- **Steps**: `GET /v3/tiers?programId=200` (includeInactive=false default)
- **Expected**:
  - HTTP 200
  - `data` contains 2 tiers, STOPPED tier absent
  - Confirms `is_active=0` rows excluded from MySQL query + STOPPED status filtered in MongoDB

---

**TS-05: List tiers with includeInactive=true**
- **Category**: Happy Path / AC-1-5
- **Pre-conditions**: Program 200 has 2 ACTIVE tiers, 1 STOPPED tier
- **Steps**: `GET /v3/tiers?programId=200&includeInactive=true`
- **Expected**:
  - HTTP 200
  - `data` contains 3 tiers (including STOPPED)
  - STOPPED tier has `memberCount=null` (cannot query `is_active=0` rows in enrollment)
  - STOPPED tier has `slabId` populated (set when it was ACTIVE)

---

**TS-06: Get single tier — ACTIVE tier with memberCount**

- **Pre-conditions**: ACTIVE tier with tierId "abc-123" in Org 100
- **Steps**: `GET /v3/tiers/abc-123` with token (orgId=100)

  - HTTP 200
  - Full TierResponse with all fields
  - `memberCount` populated (from customer_enrollment via Thrift getMemberCountPerSlab)
  - `slabId` non-null
  - `createdOn`, `lastModifiedOn` are ISO-8601 with UTC timezone

### 2.5 Idempotency Scenarios

---

**TS-43: APPROVE retry — already approved (slabId already set)**
- **Category**: Idempotency / R-8
- **Pre-conditions**: Tier was already approved (status=ACTIVE, slabId=42). Network timeout caused client to retry.
- **Steps**: `POST /v3/tiers/{id}/status` with `{"action": "APPROVE"}` — second call
- **Expected**:
  - HTTP 200 (not error)
  - Returns existing TierResponse (slabId=42, status=ACTIVE)
  - Thrift `createOrUpdateSlab` NOT called again
  - No duplicate MySQL row created

---

**TS-44: APPROVE retry — Thrift call in progress (concurrent)**
- **Category**: Idempotency / Concurrency
- **Pre-conditions**: Two APPROVE requests arrive simultaneously for same tierId
- **Steps**: Two concurrent `POST /v3/tiers/{id}/status` APPROVE requests
- **Expected**:
  - Exactly one succeeds: HTTP 200, status=ACTIVE
  - Second one returns HTTP 400 "request already processing" or HTTP 200 (idempotent response)
  - Only ONE MySQL slab row created — no duplicate
  - Version check (optimistic locking on TierDocument.version) prevents double-write

---

- **Steps**: `POST /v3/tiers` with serialNumber=3
- **Expected**:
  - HTTP 400
  - Error: serialNumber 3 already in use for this program (TierRepository findByOrgIdAndProgramIdAndSerialNumber check)

---

**TS-46: DELETE already-deleted tier**
- **Category**: Idempotency / AC-5-5
- **Pre-conditions**: Tier already in STOPPED status (previously soft-deleted)
- **Steps**: `DELETE /v3/tiers/{id}` second call

**TS-54: APPROVE — Thrift succeeds but MongoDB update fails**
- **Category**: Failure / data integrity
- **Pre-conditions**: Thrift writes MySQL row, then MongoDB save throws exception

### 2.9 Concurrency Scenarios

**TS-61: Two users approve same tier simultaneously**
- **Category**: Concurrency / G-10 / G-11.5
- **Steps**: Two threads simultaneously call `POST /v3/tiers/{id}/status` APPROVE
- **Expected**:
  - TierDocument.version check (optimistic lock) ensures only one wins
  - First write succeeds: version incremented
  - Second write detects version mismatch: returns HTTP 409 (or HTTP 200 with idempotent response if slabId already set)

  - **Evidence**: Designer phase resolved this: "APPROVE checks version before write to detect concurrent approval" (session-memory.md line 220)

**TS-62: Two users update same DRAFT tier simultaneously**
- **Category**: Concurrency / G-10
- **Pre-conditions**: Two users read same DRAFT tier at version=1, both send PUT

- **Expected**:
  - One succeeds: version increments to 2
  - Second should fail or override (last-write-wins vs optimistic lock)
  - **QUESTION FOR USER (C2)**: Does PUT use version-based optimistic locking, or is it last-write-wins? The Designer defines version field on TierDocument for concurrent APPROVE protection. Is it also enforced on PUT? Recommend requiring version field in TierRequest and rejecting stale versions.
