## Current Behaviour

### Tier Configuration (docs.capillarytech.com + codebase)

The Capillary loyalty platform uses **Tiers** (called **Slabs** internally — `ProgramSlab` entity, `program_slabs` table) as the structural backbone of every loyalty program. Current state:

- **Tier creation**: Navigate to Menu > Loyalty+ > Programs > Edit Program > Tiers tab. A default base tier exists (no conditions). Subsequent tiers are created above the highest existing tier — cannot insert between existing tiers. Cannot delete existing tiers.
- **Eligibility criteria**: 4 KPI types — current points, lifetime points, lifetime purchases, tracker value. The KPI type is set once and applies to all tiers in a program.
- **Upgrade**: 3 sequences — points-first, tier-first, or split. Implemented via `SlabUpgradeService` + `UpgradeSlabActionImpl`.
- **Validity**: 3 models — fixed duration, until registration date, until fixed date. Daily or monthly validity checks.
- **Renewal**: conditions based on purchases, visits, points, or trackers. AND/OR/Custom logic. Implemented via `RenewSlabInstructionImpl`.
- **Downgrade**: 3 target options — one tier below, matching eligibility, or lowest tier. `isDowngradeOnReturnEnabled` flag exists for return-transaction validation. Implemented via `SlabDowngradeService` + `DowngradeSlabActionImpl`.
- **No status lifecycle**: `ProgramSlab` has no `status` column. Tiers are either present or not.
- **No maker-checker for tiers**: Approval workflow exists for Promotions (`RequestManagementController` in intouch-api-v3) but is not wired for tier entity type.
- **No soft delete**: Tiers cannot be deleted at all today.
- **No REST API**: Tier CRUD is done through the legacy UI config flow, not via a self-reliant API.

### Maker-Checker Pattern (UnifiedPromotion — intouch-api-v3)

The existing maker-checker flow for promotions follows this pattern:
1. **Create** → entity created in `DRAFT` status
2. **Submit for approval** → `PUT /v3/requests/PROMOTION/{id}/status` → status becomes `PENDING_APPROVAL`
3. **Approve** → `POST /v3/unifiedPromotions/{id}/review` with `APPROVE` → status becomes `ACTIVE`
4. **Reject** → status reverts to `DRAFT` with mandatory comment
5. **Stop** → status becomes `STOPPED`

Key components: `UnifiedPromotionController`, `UnifiedPromotionFacade`, `EntityType` enum, `PromotionStatus` enum, `RequestManagementFacade`, `ResponseWrapper<T>` for structured responses, `TargetGroupErrorAdvice` for field-level validation errors.

## Problem Statement

Loyalty tiers are the most consequential configuration in any loyalty program, but there is no self-reliant API to manage them. Today, tier configuration is only accessible through the legacy UI flow, which is fragmented across multiple screens and inaccessible to programmatic consumers (including the future aiRa conversational AI). Program managers spend ~45 minutes creating a single tier because the configuration is scattered and there is no validation feedback, no approval workflow, and no ability to deactivate tiers.

The platform needs a complete set of Tier CRUD APIs that:
- Are self-reliant (validation in API, not in frontend)
- Follow the established maker-checker pattern (same as UnifiedPromotion)
- Return structured field-level errors (future-proofed for aiRa consumption)
- Support the full tier lifecycle (Draft → Pending Approval → Active → Stopped)
- Allow soft deletion of tiers

---

## Goals

1. **API-first tier management**: Complete CRUD operations for tiers via REST APIs in intouch-api-v3, with Thrift backend in emf-parent.
2. **Maker-checker approval**: Every tier create/update goes through the approval workflow, following the UnifiedPromotion pattern.
3. **Self-reliant validation**: APIs return structured, field-level validation errors — not generic 500s. Same validation that a UI would enforce.
4. **Tier lifecycle**: Support Draft → Pending Approval → Active → Stopped status transitions.
5. **Soft delete**: Allow deactivation of tiers via an `active` flag without hard-deleting data.

## Scope

### In Scope

| Item | Details |
|------|---------|
| **GET /tiers** | List all tiers for a program with full configuration |
| **GET /tiers/{tierId}** | Get single tier with full configuration |
| **POST /tiers** | Create a new tier with field-level validation. Creates in DRAFT status. |

| **Thrift service layer** | New Thrift definitions for tier CRUD in emf-parent |
| **Service + DAO layer** | Backend persistence in emf-parent |

## User Stories

### US-1: List Tiers
**As a** program manager (via API consumer),
**I want to** retrieve all tiers for a program with their full configuration,
**so that** I can understand the current tier hierarchy at a glance.

**Acceptance Criteria:**
- [ ] GET /tiers returns all active tiers for the authenticated org's program
- [ ] Response includes: tier name, description, color, eligibility KPI type, threshold, validity period, renewal conditions, downgrade target, downgrade-on-return flag, upgrade schedule, upgrade bonus, status
- [ ] Tiers are ordered by hierarchy (base tier first, highest tier last)
- [ ] Soft-deleted tiers (active=0) are NOT returned by default
- [ ] Optional query parameter `includeInactive=true` returns all tiers including soft-deleted
- [ ] Response uses `ResponseWrapper<T>` format

### US-2: Get Single Tier
**As a** program manager (via API consumer),
**I want to** retrieve the full configuration of a specific tier,
**so that** I can inspect or edit its details.

**Acceptance Criteria:**
- [ ] GET /tiers/{tierId} returns full configuration for the specified tier
- [ ] Returns 404 (via ResponseWrapper error) if tier does not exist or is soft-deleted
- [ ] Response includes all fields from US-1

### US-3: Create Tier
**As a** program manager (via API consumer),
**I want to** create a new tier with full validation,
**so that** I can extend the tier hierarchy without errors.

**Acceptance Criteria:**
- [ ] POST /tiers creates a new tier in DRAFT status
- [ ] Required fields validated: name (non-blank), eligibility KPI type, eligibility threshold (> 0), validity period, downgrade target (or "None" for base tier)
- [ ] Optional fields accepted: description, color, upgrade schedule, renewal condition, renewal schedule, upgrade bonus, downgrade-on-return flag
- [ ] Field-level validation errors returned in `ResponseWrapper.errors[]` format: `{code, message}` per invalid field
- [ ] New tier is added above the highest existing tier (cannot insert between)
- [ ] Eligibility KPI type must match the program's existing KPI type (if tiers already exist)
- [ ] Threshold must be higher than the current highest tier's threshold
- [ ] If maker-checker is enabled for the org, tier is created in DRAFT (not ACTIVE)

### US-4: Update Tier
**As a** program manager (via API consumer),
**I want to** update any configuration field of an existing tier,
**so that** I can adjust tier settings as program strategy evolves.

**Acceptance Criteria:**
- [ ] PUT /tiers/{tierId} updates the specified tier's configuration
- [ ] Same field-level validation as create (US-3)
- [ ] Cannot change eligibility KPI type (set once per program)
- [ ] If maker-checker is enabled, update creates a new DRAFT version (does not modify ACTIVE directly)
- [ ] Returns 404 if tier does not exist or is soft-deleted
- [ ] If tier is in PENDING_APPROVAL status, update is rejected with error (must approve/reject first)

### US-5: Soft Delete Tier
**As a** program manager (via API consumer),
**I want to** deactivate a tier I no longer need,
**so that** it is hidden from active use without losing historical data.

**Acceptance Criteria:**
- [ ] DELETE /tiers/{tierId} sets `active=0` on the tier record
- [ ] Cannot soft-delete the base tier (returns validation error)
- [ ] Cannot soft-delete a tier that is the downgrade target of another active tier (returns validation error listing dependent tiers)
- [ ] Soft-deleted tier is excluded from GET /tiers default response
- [ ] Soft-deleted tier's members need to be addressed — either migrated or left in place (decision: leave in place, mark for re-evaluation at next cycle)
- [ ] Returns 404 if tier does not exist or is already soft-deleted

### US-6: Submit Tier for Approval
**As a** program manager (via API consumer),
**I want to** submit a draft tier configuration for approval,
**so that** a checker can review before it goes live.

**Acceptance Criteria:**
- [ ] PUT /v3/requests/TIER/{tierId}/status with action SUBMIT_FOR_APPROVAL
- [ ] Status transitions from DRAFT → PENDING_APPROVAL
- [ ] Only DRAFT status tiers can be submitted
- [ ] Uses existing `RequestManagementController` with new `TIER` entity type added to `EntityType` enum
- [ ] Notification sent to approver (follow existing promotion notification pattern)

### US-7: Approve / Reject Tier
**As a** platform admin / approver (via API consumer),
**I want to** approve or reject a pending tier configuration,
**so that** only reviewed changes go live.

**Acceptance Criteria:**
- [ ] POST /tiers/{tierId}/review with `approvalStatus: APPROVE` or `REJECT`
- [ ] APPROVE: status transitions PENDING_APPROVAL → ACTIVE
- [ ] REJECT: status transitions PENDING_APPROVAL → DRAFT, mandatory comment required
- [ ] Only PENDING_APPROVAL tiers can be approved/rejected
- [ ] On APPROVE: tier configuration becomes effective (visible in program, affects member evaluations)
- [ ] Follows same review pattern as `UnifiedPromotionController.reviewPromotion()`

## Constraints

- **Tier naming**: "Tier" in API surface, "Slab" internally in codebase (`ProgramSlab`, `program_slabs`)
- **KPI type immutable**: eligibility KPI type is set once for a program's first tier and cannot be changed for subsequent tiers
- **Sequential tiers**: new tiers added above the highest existing tier only — no insertion between
- **Active flag default**: new `active` column in `program_slabs` defaults to 1 (true)
- **UnifiedPromotion pattern**: controller in intouch-api-v3, Thrift + service in emf-parent
- **ResponseWrapper format**: all responses use existing `ResponseWrapper<T>` with `{data, errors, warnings}`
- **Field-level validation**: use existing `TargetGroupErrorAdvice` (global `@ControllerAdvice`) — Bean Validation annotations on DTOs

## Assumptions

- A1: The existing `EntityType` enum in intouch-api-v3 can be extended with `TIER` without breaking existing consumers [C6 — enum addition is backward-compatible]
- A2: The `PromotionStatus` enum values (DRAFT, PENDING_APPROVAL, ACTIVE, STOPPED) are reusable for tiers, or a new `TierStatus` enum with the same values is created [C5 — architectural decision for Phase 6]
- A3: Partner program tier sync (`PartnerProgramTierSyncConfiguration`) is not affected by the new CRUD APIs since we're adding an API layer above existing backend logic [C4 — needs verification during codebase research]
- A4: Existing tier evaluation logic (upgrade/downgrade/renewal) continues to work unchanged — the new APIs only manage configuration, not evaluation execution [C6 — CRUD is separate from evaluation]
- A5: `TierConfiguration` DTO fields (`dailyDowngradeEnabled`, `retainPoints`, `slabConfigs`) are exposed in the new API alongside the individual tier fields [C4 — depends on whether these are per-tier or per-program settings]
