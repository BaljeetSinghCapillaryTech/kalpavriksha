Tier CRUD REST APIs with Maker-Checker

⚠ CONDITIONAL PASS — 1 deferred operational blocker (DDL pending lead approval), 1 accepted risk (APPROVE reverse failure gap)

Executive Summary

This pipeline delivered a complete Tier CRUD REST API for the Capillary loyalty platform, enabling programmatic management of tier (slab) configuration with a full maker-checker approval workflow. The implementation follows a MongoDB-first architecture — DRAFT and PENDING_APPROVAL tiers live exclusively in MongoDB, while only ACTIVE tiers are synced to MySQL via Thrift on APPROVE, ensuring the existing evaluation engine (upgrade / downgrade / renewal) remains completely unaffected.

35 requirements were traced end-to-end across 14 pipeline phases. 34 passed fully and 1 is PARTIAL (REQ-34: partner sync error message quality — constraint is enforced at the right layer but the error message prefix could be cleaner). The feature spans 4 repositories (intouch-api-v3, emf-parent, thrifts, cc-stack-crm), introduces 6 new REST endpoints, and plans 26 test methods (11 unit + 15 integration) as the next implementation step.

Two items are deferred: the is_active DDL ALTER TABLE (pending lead approval in cc-stack-crm) and the APPROVE reverse-failure gap (accepted risk — identical to the UnifiedPromotion pattern already in production). All 6 ADRs are implemented as designed. All critical guardrails (G-01 timezone, G-03 security, G-07 multi-tenancy, G-12 AI-specific) are compliant.

API Contract

All endpoints are prefixed /v3/tiers. Authentication: AbstractBaseAuthenticationToken (Bearer token). Response format: ResponseWrapper<T> with {data, errors[], warnings[]}.

MethodPathPurposeRequest BodySuccess Response
GET /v3/tiers?programId={id}&includeInactive=false List all tiers for a program. Default excludes STOPPED/SNAPSHOT. — 200 ResponseWrapper<List<TierResponse>>
GET /v3/tiers/{tierId} Get single tier. Returns STOPPED tiers. 404 only if is_active=0 in MySQL. — 200 ResponseWrapper<TierResponse>
POST /v3/tiers Create tier in DRAFT status. MongoDB only. TierRequest (name, programId, serialNumber, eligibility, validity, renewal, downgrade, upgradeBonus) 201 ResponseWrapper<TierResponse>
PUT /v3/tiers/{tierId} Update DRAFT in-place. If ACTIVE, creates a new DRAFT with parentId (copy-on-write). TierRequest (same as create) 200 ResponseWrapper<TierResponse>
DELETE /v3/tiers/{tierId} Soft delete — sets is_active=0 in MySQL via Thrift. Validates: not base tier, no members, not downgrade target, not in partner sync. — 200 ResponseWrapper<null>
POST /v3/tiers/{tierId}/status Status transitions: SUBMIT_FOR_APPROVAL, APPROVE, REJECT, STOP TierStatusRequest (action, comment) 200 ResponseWrapper<TierResponse>
