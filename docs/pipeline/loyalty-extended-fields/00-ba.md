# Business Analysis ‚Äî Loyalty Extended Fields CRUD
> Feature: Loyalty Extended Fields CRUD
> Ticket: loyaltyExtendedFields (Jira: CAP-183124)
> Phase: BA + PRD (Phase 1)
> Date: 2026-04-22

---

## Current Behaviour

The platform already has extended fields for customer, transaction, and lineitem entity types ‚Äî managed via the CDP team's `extended_fields` table in the meta DB. The loyalty team has **no ownership** of that table.

In the subscription module (intouch-api-v3), a preliminary `SubscriptionProgram.ExtendedField` model exists in MongoDB (`subscription_programs` collection) with fields: `type` (currently mis-named as `ExtendedFieldType` with wrong values `CUSTOMER_EXTENDED_FIELD`/`TXN_EXTENDED_FIELD`), `key`, `value`. This model was implemented (tests BT-EF-01 to BT-EF-06 exist) but **has no CRUD APIs, no business logic, and no validation framework**.

There is no loyalty-owned registry that defines which custom fields are allowed per org and scope. Any extended field values passed into subscription programs are currently persisted without validation.

---

## Problem Statement

1. **No loyalty-owned EF registry** ‚Äî the loyalty team cannot define or govern custom field metadata independently of CDP.
2. **No validation** ‚Äî callers can pass arbitrary `key`/`value` pairs in `extendedFields` on subscription programs; nothing checks them against a registry.
3. **Wrong model** ‚Äî `ExtendedFieldType` enum (`CUSTOMER_EXTENDED_FIELD`, `TXN_EXTENDED_FIELD`) misrepresents the concept; the field is actually a `scope` discriminator.
4. **No CRUD APIs** ‚Äî org admins have no self-serve mechanism to configure custom fields for their subscription programs.

---

## Goals

1. Create a loyalty-owned extended field config registry (`loyalty_extended_fields` MySQL table in `warehouse` DB).
2. Expose CRUD APIs so org admins can define, update, deactivate, and list custom field definitions per scope.
3. Implement a validation framework: when a subscription program is created or updated with `extendedFields`, validate each entry against the registry.
4. Fix the `SubscriptionProgram.ExtendedField` model ‚Äî rename `type` ‚Üí `scope`, replace wrong enum with correct scope values.

## User Stories

### Epic 1 ‚Äî EF Config Registry (CRUD)

**EF-US-01** ‚Äî Create EF Definition
> As an org admin, I want to define a new custom extended field (name, data type, scope, mandatory flag, default value) for scope `SUBSCRIPTION_META`, so that subscription programs in my org can carry that custom attribute.

Acceptance Criteria:
- `POST /v3/extendedfields/config` creates a row in `loyalty_extended_fields` for the caller's org
- Uniqueness: `(org_id, scope, name)` ‚Äî duplicate returns `409 CONFLICT`
- `scope` must be a valid allowed value; invalid scope returns `400`
- `data_type` must be one of: `STRING`, `NUMBER`, `BOOLEAN`, `DATE`; invalid returns `400`
- `status` defaults to `ACTIVE`
- `created_on` and `last_updated_on` populated automatically (UTC)
- Returns the created EF config with its `id`

**EF-US-02** ‚Äî Update EF Definition
> As an org admin, I want to update an existing EF definition (default value, is_mandatory, label/description) so that I can refine the field config without recreating it.

Acceptance Criteria:
- `PUT /v3/extendedfields/config/{id}` updates the matching row for the caller's org
- `name` and `scope` are immutable after creation (cannot be changed via PUT)
- `data_type` is immutable after creation
- Non-existent `id` or wrong `org_id` ‚Üí `404 NOT FOUND`
- `last_updated_on` updated on every successful PUT (UTC)

**EF-US-03** ‚Äî Deactivate EF Definition (Soft Delete)
> As an org admin, I want to deactivate an extended field so that it is no longer available for use without losing its history.

Acceptance Criteria:
- `DELETE /v3/extendedfields/config/{id}` sets `status = INACTIVE` (soft delete ‚Äî no row deletion)
- Non-existent `id` or wrong `org_id` ‚Üí `404 NOT FOUND`
- Already inactive ‚Üí `409 CONFLICT` or idempotent success (Architect to decide)
- Deactivated field config no longer passes validation when consumed

**EF-US-04** ‚Äî List EF Definitions
> As an org admin, I want to list all extended field definitions for my org, filterable by scope and status, so that I can see what is configured.

Acceptance Criteria:
- `GET /v3/extendedfields/config` returns all EF configs for the caller's `org_id`
- Supports query params: `?scope=SUBSCRIPTION_META`, `?status=ACTIVE`
- Returns paginated response (page + size params) per G-04.2
- Empty result returns `200` with empty list (not `404`)

---

### Epic 2 ‚Äî EF Validation on Subscription Programs

**EF-US-05** ‚Äî Validate EF values on Subscription Create
> As a developer integrating with the subscription API, I want the system to validate extended fields on `POST /v3/subscriptions` so that invalid field names, wrong data types, or missing mandatory fields are rejected with a clear error.

Acceptance Criteria:
- When `extendedFields` is provided on subscription create, for each entry:
  - `scope` must be `SUBSCRIPTION_META` (for now)
  - `key` must match an `ACTIVE` EF definition in `loyalty_extended_fields` for `(org_id, scope)` ‚Üí else `400`
  - `value` data type must match the registered `data_type` ‚Üí else `400`
- All mandatory `ACTIVE` EF definitions for `(org_id, SUBSCRIPTION_META)` must be present ‚Üí else `400`
- If `extendedFields` is `null` or omitted AND mandatory fields exist ‚Üí `400`
- Valid `extendedFields` ‚Üí persisted in MongoDB `subscription_programs.extendedFields`

**EF-US-06** ‚Äî Validate EF values on Subscription Update
> As a developer, I want EF validation to also apply on `PUT /v3/subscriptions/{id}` so that updates remain consistent with the registry.

Acceptance Criteria:
- Same validation rules as EF-US-05 when `extendedFields` is provided
- If `extendedFields` is `null` on PUT ‚Üí **preserve existing values** (existing null-guard behaviour, R-33 from prior subscription work)
- If `extendedFields` is an empty list `[]` ‚Üí clear all EF values (explicit intent to remove)

---

### Epic 3 ‚Äî Model Correction

**EF-US-07** ‚Äî Rename `ExtendedFieldType` ‚Üí scope
> As a developer, I want `SubscriptionProgram.ExtendedField.type` renamed to `scope` and the wrong `ExtendedFieldType` enum removed so that the codebase accurately represents the domain.

Acceptance Criteria:
- `ExtendedFieldType` enum (`CUSTOMER_EXTENDED_FIELD`, `TXN_EXTENDED_FIELD`) is deleted
- `SubscriptionProgram.ExtendedField.type` field renamed to `scope` (type: `String`)
- Tests BT-EF-01 to BT-EF-06 updated to use `SUBSCRIPTION_META` instead of `CUSTOMER_EXTENDED_FIELD`/`TXN_EXTENDED_FIELD`
- MongoDB field name in documents updated (`type` ‚Üí `scope`) or migration handled
- No other callers of `ExtendedFieldType` broken (grep confirms no other usages)

## Constraints

| Constraint | Source |
|-----------|--------|
| `loyalty_extended_fields` lives in `warehouse` DB, owned by loyalty team | Architecture decision |
| `scope` column is `VARCHAR` (not MySQL ENUM) ‚Äî future scope values will grow too large for DB ENUM | Design decision |
| Application-level scope validation enforces allowed values (enum-like, extensible) | Design decision |
| Only `SUBSCRIPTION_META` scope in scope for Sprint 1-3 | User decision |
| V3 ‚Üí EMF communication via Thrift (new methods in `EMFService`) | Architecture decision |
| EMF owns warehouse DAO; V3 has no direct warehouse DB access | Architecture constraint (verified) |
| cc-stack-crm schema: CREATE TABLE only, no ALTER/Flyway ‚Äî PR merge deploys | Project convention (C7) |
| Soft deletes only (`status = INACTIVE`) ‚Äî no physical row deletion | G-05.5, BRD |
| Uniqueness: `(org_id, scope, name)` at DB + service layer | BRD, G-05.3 |
| All timestamps in UTC | G-01.1 |
| All queries must include `org_id` filter | G-07.1 |
| Mandatory fields must be validated at V3 boundary | G-03.2 |

## Assumptions

| # | Assumption |
|---|-----------|
| A-01 | `status` column will use `ACTIVE`/`INACTIVE` VARCHAR per BRD, not `is_active tinyint` from cc-stack-crm convention ‚Äî Architect to confirm |
| A-02 | Org-level config (max EF count per org) ‚Äî stored in existing `program_config_key_values` or a lightweight config entry; Architect to decide |
| A-03 | `value` is always stored as `String` in MongoDB; data type validation is application-level only (same pattern as existing `ExtendedFieldsData` in Thrift) |
| A-04 | `SUBSCRIPTION_META` is the only scope value validated at application level for Sprint 1-3; other values return `400` |
| A-05 | `DELETE` is idempotent ‚Äî deleting an already-INACTIVE field returns success (not 409); Architect to confirm |
| A-06 | Existing `SubscriptionProgram.extendedFields` MongoDB field name stays as `extendedFields` ‚Äî only the Java field `type` is renamed to `scope`; the JSON key is updated to `scope` |
