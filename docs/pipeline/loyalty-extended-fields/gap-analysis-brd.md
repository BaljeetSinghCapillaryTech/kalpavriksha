## GUARDRAILS Compliance

| Guardrail | Rule | Design Compliance | Notes |
|-----------|------|------------------|-------|
| G-07 | `org_id` in every query | **COMPLIANT (proposed)** | PRD table DDL includes `org_id BIGINT NOT NULL` with composite index `idx_org_scope_active (org_id, scope, is_active)`. Thrift request structs include `orgId`. NFR section explicitly states "every DB query must include `org_id` filter". Existing subscription pattern confirmed: V3 sources `org_id` from `user.getOrgId()` (auth context), not request body. |
| G-01 | UTC timestamps stored and returned | **AT RISK** | PRD table DDL defines `created_on DATETIME NOT NULL` and `last_updated_on DATETIME NOT NULL` — **DATETIME has no timezone awareness in MySQL**. cc-stack-crm convention for the closely analogous `custom_fields.sql` uses `created_on DATETIME` + `auto_update_time TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`. TIMESTAMP type stores in UTC and converts on retrieval; DATETIME does not. The Thrift response struct returns timestamps as `string` (ISO-8601), but the DB column type must be TIMESTAMP or the application must explicitly write UTC values. **Recommendation**: use `TIMESTAMP` not `DATETIME` for `created_on` and `last_updated_on`, consistent with cc-stack-crm warehouse tables (e.g., `alternate_currencies.sql`, `capping_config.sql`). |
| G-09.5 | New Thrift fields must be `optional` for backward compatibility | **VIOLATION (proposed design)** | The PRD-proposed `LoyaltyExtendedFieldConfig` struct (the response struct) declares fields 1–8 all as `required`: `required i64 id`, `required i64 orgId`, `required string name`, `required string scope`, `required string dataType`, `required bool isMandatory`, `required bool isActive`, `required string createdOn`, `required string lastUpdatedOn`. **The NFR section itself states "New Thrift fields must be `optional` for backward compatibility (G-09.5)"** — yet the struct definitions contradict this by using `required` throughout. Thrift `required` fields cannot be added to an existing struct in a backward-compatible way. For new structs (not added fields), `required` does not cause wire breakage, but it prevents future evolution: any client that receives this struct must provide all required fields or Thrift will reject the message. **Corrective action**: all fields in new structs should be `optional` except the primary key and any field where absence truly indicates a protocol error. At minimum, `defaultValue`, `createdOn`, `lastUpdatedOn` should be `optional`. |
| G-03 | `org_id` from auth context, not request body | **COMPLIANT (existing pattern)** | Confirmed from `SubscriptionController.java`: org_id is always sourced from `user.getOrgId()` (authenticated user object), never from the request body. The Thrift `orgId` field in `CreateLoyaltyExtendedFieldRequest` will be populated by the V3 facade from auth context — this pattern is already established. PRD NFR confirms: "All endpoints require org-level authentication; org_id from auth context, never from request body." |
| G-05.5 | Soft deletes (`is_active=0`) consistent with codebase conventions | **COMPLIANT** | PRD DDL uses `is_active TINYINT(1) NOT NULL DEFAULT 1` (D-14). This exactly matches the `custom_fields.sql` pattern (`is_active tinyint(1) NULL DEFAULT '1'`) and program_config_key_values (`is_valid tinyint(1)`). The BA originally proposed `status VARCHAR(20)` but this was corrected to `is_active TINYINT(1)` in GQ-01. Convention alignment confirmed. |

## Missing from BA/PRD

The following items are revealed by the code that the BA/PRD did not mention or left ambiguous:

| # | Category | Finding | Impact |
|---|----------|---------|--------|
| M-01 | **Schema convention mismatch — timestamp** | cc-stack-crm warehouse tables universally use `auto_update_time TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` as the auto-managed update timestamp. The PRD proposes `last_updated_on DATETIME NOT NULL` which breaks convention, requires manual application-side updates, and has no timezone awareness. The Architect must decide: adopt `auto_update_time TIMESTAMP` (convention) or keep `last_updated_on DATETIME` (application-managed, aligns with PRD response field name). | Medium — affects schema portability and G-01 compliance |
| M-02 | **`program_config_key_values` is program-scoped, not org-scoped** | The table has a mandatory `program_id` column. D-15 says "org-level max EF count stored in `program_config_key_values`" — but there is no way to store a purely org-level key without a program_id. The `ProgramConfigKeyValue` entity and DAO are keyed by `(org_id, program_id, key_id)`. Storing an org-wide setting requires either a sentinel `program_id` (e.g., 0 or -1) or a `ProgramConfigKey` named something like `MAX_LOYALTY_EXTENDED_FIELDS_PER_ORG`. Neither approach is defined in the BA/PRD. | High — D-15 is blocked until this is resolved |
| M-03 | **BT-EF-05 description contradicts test assertion** | The class Javadoc says "BT-EF-05: extendedFields NOT copied on duplicateSubscription (fresh DRAFT)" but the test asserts they ARE copied from source. The `SubscriptionFacade.duplicateSubscription` at line 385 copies extendedFields. The BA's intent comment is wrong; the test and facade are correct. This must be clarified before SDET writes new tests referencing BT-EF-05 semantics. | Low — causes confusion in future test authoring |
| M-04 | **No `auto_update_time` audit column in proposed DDL** | The proposed DDL has `created_on` and `last_updated_on` but no `auto_update_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`. Every other cc-stack-crm warehouse table that has a `last_updated_on` also has `auto_update_time` (e.g., `alternate_currencies.sql`, `badges_earned_stats.sql`, `capping_config.sql`, `benefits_awarded_stats.sql`). The convention is `auto_update_time` is MySQL-managed (for replication lag tracking / ETL); `last_updated_on` or `created_on` is application-managed. The DDL as proposed omits the MySQL-managed column. | Medium — deviates from warehouse replication/ETL convention |
| M-05 | **Thrift `required` vs `optional` conflict with G-09.5** | The PRD's own NFR section states G-09.5 (new Thrift fields must be optional) but the proposed Thrift structs use `required` for almost all fields in `LoyaltyExtendedFieldConfig` and `CreateLoyaltyExtendedFieldRequest`. These are new structs (not modifications to existing ones), so wire breakage is not an immediate concern — but the mismatch between stated policy and actual spec will cause the `/backend-readiness` skill to flag a violation at Phase 10b. The Architect/Designer must decide: declare all new struct fields `optional` (safest for future extension) or document a waiver to G-09.5 for new-struct initial fields. | Medium — will be a blocker at backend-readiness gate |
| M-06 | **`ExtendedFieldsData` struct already exists with a different semantics** | `emf.thrift:655` has `ExtendedFieldsData { name, value, previousValue }` — this is event-time EF pair data, not config. The BA says "ExtendedFieldsData struct exists" as supporting evidence for the Thrift integration, but misrepresents what it does. The new `LoyaltyExtendedFieldConfig` is a different concept entirely and must be added as a new struct alongside the existing one. No naming conflict exists, but the BA's framing can mislead implementers into thinking there is existing Thrift infrastructure for EF config — there is not. | Low — clarity risk for developers |
| M-07 | **No Thrift service evolution plan for EMFService** | `emf.thrift` has 1883 lines. Adding new service methods to `EMFService` is backward-compatible (new methods can be added without breaking existing clients) — but the BA/PRD does not acknowledge whether existing generated Thrift client stubs in V3 will need to be regenerated, or what the deployment order constraint is (EMF must be deployed before V3 calls the new methods). This is a release-coordination gap. | Medium — deployment sequencing risk |
| M-08 | **No mention of `ProgramConfigKey` lookup required** | `program_config_key_values` stores values against a `key_id` foreign key to a `program_config_keys` table (ProgramConfigKey entity). To store a new org-level EF max count, a new `ProgramConfigKey` entry (e.g., `MAX_LOYALTY_EXTENDED_FIELDS`) must first exist in the `program_config_keys` table. This lookup table is not mentioned in the BA/PRD. If this key doesn't exist, the org-level config feature (D-15) cannot be implemented using this table without a data migration. | High — may require a seed data migration in cc-stack-crm |
